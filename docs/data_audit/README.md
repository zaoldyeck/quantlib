# 全資料鏈稽核(2026-07-22 起)

**目的**(使用者定調):這個專案早期的資料結構是人工一筆一筆核對出來的,尤其財報
是非結構化來源,難免有疏漏。**從原始檔到回測可用的結構化資料,整條鏈路全部複核。**
資料錯了,上面所有回測與策略結論都不成立。

## 為什麼要有這份帳本

稽核範圍是 **92,301 個原始檔 / 24 張 cache 表 / 20 個解析函式 / 16 個 SQL view**。
一次做不完,而且中途一定會被額度或時間打斷。所以:

- **每個稽核單位的結論即時落盤**(`findings/<unit>.md` + `findings.jsonl`),
  不留在對話裡——對話會消失,repo 不會。
- **進度記在 `_progress.jsonl`**,重跑自動跳過已完成的單位。
- 中斷後接續 = 重跑同一個 workflow;已完成的 agent 呼叫由 journal 快取秒回。

**判準**:任何一次中斷都不得讓已查出的結果消失。「功虧一簣」在這裡是不可接受的
失敗模式。

## 稽核維度(三條並行)

| 維度 | 問什麼 | 風險等級 |
|---|---|---|
| **A 解析正確性** | 原始檔的每一欄,有沒有正確落到 DB?欄位錯位、單位(元 vs 千元)、編碼、TWSE 欄位漂移、正負號 | 🔴 最高——錯了整個回測是假的 |
| **B 財務定義與算式** | Slick schema、7 個 matview、9 個 view 的算式對不對?**F-Score 是否符合 Piotroski (2000) 學理** | 🔴 高——因子定義錯 = 策略選錯股 |
| **C 一致性與缺漏** | DuckDB cache 與 PG 是否逐欄一致?日期覆蓋有沒有洞?缺的補下載 | 🟠 中——缺料會讓閘門 fail-closed 靜靜砍掉候選 |

## 結論分級

- `🔴 BUG` — 確認的資料錯誤,必須修(附重現方式)
- `🟡 SUSPECT` — 可疑但需更多證據
- `🟢 OK` — 已驗證正確(**負結果也要落盤**,否則下次有人會重查一遍)
- `⚪ REAL` — 看起來像錯,但查證後是真實現象(如金融業負營收 = 匯兌損失)
