# A-stock_per_pbr_dividend_yield — 解析正確性稽核

- 原始檔:14,762 個(`data/stock_per_pbr_dividend_yield/{twse,tpex}/<year>/`)
- 解析程式:`src/main/scala/reader/TradingReader.scala:700-761`
  (`readStockPER_PBR_DividendYield`)
- 落地表:PostgreSQL `stock_per_pbr_dividend_yield`(7,738,937 列)/
  DuckDB cache `stock_per_pbr`
- 稽核日:2026-07-22
- 判定:**BUG**

## 一句話結論

**現在資料庫裡的本益比、股價淨值比、殖利率,數字本身可以信——我把 14,762 個原始檔
用另寫的解析器全部重讀一遍,7,738,937 列、四個欄位逐格比對,零差異。不能信的是
「日期」和「有沒有」:有 19 個日子存的是別天的資料(其中 2025-08-12、2026-02-25
這兩個近期交易日整天存的是 2017-12-18 的舊估值),另有 13 個真交易日整天沒資料。
還有一顆未爆彈:解析程式現在的欄位對照表是錯的,只要有人重匯一次歷史,19 年的
股價淨值比會整批變成空值或錯位——目前沒爆,只是因為「日期已在庫就跳過」擋住了。**

## 做了什麼

1. 讀 `StockPER_PBR_DividendYieldSetting`(TWSE 起 2005-09-02、TPEx 起 2007-01-02)
   與 reader 全文,寫下它宣告的欄位順序與欄數分派規則。
2. 另寫一支**用中文標頭定位欄位**的獨立解析器(不呼叫受測程式),把 14,762 個檔
   全部重讀成 7,740,350 列。
3. 與 PostgreSQL 全表 join,逐列逐欄比對 `company_name / pe / pb / dy`。
4. 用專案實際使用的 `scala-csv_2.13-1.3.6` 直接跑欄位切分,量出 TWSE 各世代真正的
   欄數(6 / 8 / 9),再套回 reader 的 `values.size match` 推算現行程式的實際輸出。
5. 檔名日期 vs 檔案內容日期全檔比對 + 每日內容指紋去重。
6. 殖利率 PIT 檢驗:`殖利率/100 × 當日收盤` 對 `ex_right_dividend` 的實際配息。
7. 覆蓋度、極端值、NULL 來源、cache↔PG 一致性。

腳本全部在 `docs/data_audit/scripts/A-stock_per_pbr_dividend_yield/`,可重跑。

## 逐項結論

### 🔴 BUG 1 — 19 個日期存的是別天的資料(16,447 列),含 2 個近期交易日

TWSE 的 bwibbu_d 端點在某些情況下會回一份**固定的 2017-12-18 快照**(標題就寫
「106年12月18日」),爬蟲照單全收、reader 用**檔名**當日期寫進資料庫,於是那天的
估值就被別天的數字蓋掉。

台積電 2026-02-25 的實際情形:

| 日期 | 收盤 | 本益比 | 股價淨值比 | 殖利率 |
|---|---|---|---|---|
| 2026-02-24 | 1965 | 32.11 | 10.20 | 1.12 |
| **2026-02-25** | **2015** | **17.11** | **4.11** | **3.08** ← 這是 2017-12-18 的值 |
| 2026-02-26 | 1995 | 30.12 | 9.55 | 1.10 |

便宜股。同一天 905 檔全部中招。

### 🔴 BUG 2 — reader 現行的 TWSE 欄位對照表是錯的,只是還沒被觸發

`TradingReader.scala:734-738`:

```scala
values.size match {
  case 6 => (…, values(2), values(5), values(3))   // pe, pb, dy
  case 7 => (…, values(4), values(5), values(2))
  case _ => (…, values(5), values(6), values(3))
}
```

用專案實際的 `scala-csv 1.3.6` 量出來的真實欄位(`06_ParseProbe.java`):

```
era1 "1101","台泥","16.92","5.91","1.07",\r\n
   -> size=6 [ (0)=1101 (1)=台泥 (2)=16.92 (3)=5.91 (4)=1.07 (5)='' ]
era2 "1101","台泥","4.12","105","20.47","1.22","105/4",\r\n
   -> size=8 [ (0)=1101 (1)=台泥 (2)=4.12 (3)=105 (4)=20.47 (5)=1.22 (6)=105/4 (7)='' ]
era3 "1101","台泥","33.55","2.98","112","30.23","1.09","113/1",\r\n
   -> size=9 [ (0)…(5)=30.23 (6)=1.09 (7)=113/1 (8)='' ]
```

對照後現行程式的實際輸出:

| 世代 | 欄數 | 走哪個 case | 本益比 | 股價淨值比 | 殖利率 |
|---|---|---|---|---|---|
| 2005-09 ~ 2017-04 | 6 | `case 6` | ✅ | ❌ `values(5)` = 空字串 → **NULL**(應為 `values(4)`) | ✅ |
| 2017-04 ~ 2024-06 | 8 | `case _` | ❌ 取到**股價淨值比** | ❌ 取到「105/4」→ **NULL** | ❌ 取到**股利年度** → **105.0**(當成 105% 殖利率) |
| 2024-07 ~ 今 | 9 | `case _` | ✅ | ✅ | ✅ |
| `case 7` | — | **永不觸發**(TWSE 沒有 7 欄型態) | 死碼 | | |

具體:2017-05-02 台泥若現在重匯,會變成 `pe=1.22, pb=NULL, dy=105.0`(正確值
`pe=20.47, pb=1.22, dy=4.12`);2012-03-01 台泥會變成 `pb=NULL`(正確 1.47)。

**為什麼現在沒爆**:`TradingReader.scala:706` 用「`(market, 檔名)` 已在資料庫就跳過」
去重,舊日期永遠不會被重讀。commit `b14f115`(2026-04-23)為了修 2024-07 新增
「收盤價」欄的錯位而改了這段,**修好了第三世代、同時弄壞了前兩世代**,只是被
去重規則遮住。任何一次「刪表重匯」或「刪某天重讀」都會立刻炸開 2005-2024 共
約 387 萬列。程式碼上方的註解(`TradingReader.scala:726-733`)三行描述**全部與
實測不符**,是這個錯誤的來源。

### 🔴 BUG 3 — 13 個真交易日整天沒有估值資料

| 市場 | 日期 | 原始檔 |
|---|---|---|
| twse | 2008-08-26, 2009-12-12, 2014-05-07, 2016-09-09, 2016-11-01, 2016-12-21, 2017-02-13, 2017-02-24, 2020-02-06, 2025-08-05, 2026-04-01 | 存在但只有 2 bytes(`\r\n`,空回應被當成休市) |
| tpex | 2010-11-22, 2016-08-31 | 檔案根本不存在 |

這些日子 `daily_quote` 都有幾百到一千多檔的報價,所以確定是交易日。空回應被當成
「這天沒開市」永久留檔,不會重試;缺這些日子的估值,任何用 `stock_per_pbr` 當閘門
的策略在這幾天會靜靜把候選砍光(和 2026-07-15 Evergreen 事故同一機制)。

### ⚪ REAL — TPEx 2007-2010 原始檔的「股利年度/每股股利」是抓取當下的值(前視),
但**沒有進資料庫**,殖利率本身是乾淨的

TPEx 2007-01 ~ 2010-12 的原始檔裡,508,178 列的「股利年度」是**民國 108(2019)**
——比檔案日期晚 9~12 年,不可能在當時知道。乍看是重大前視污染。

實測結果:**污染只在那兩個沒有被匯入的欄位上**。例:2007-08-15 皇田 9951
原始列為 `"9951","皇田","12.75","3.50000000","108","4.50","3.11"` —— 每股股利欄
寫 3.50(2019 年的股利)、股利年度寫 108,但殖利率 4.50% × 當日收盤 55.6 = 2.502,
正是當時真實的 2.5 元配息。恩得利 1333 同日每股股利欄為 0 而殖利率 12.84%,
兩欄根本不相干。

全庫驗證(`05_yield_pit_validation.py`):`殖利率/100 × 當日收盤` 對
`ex_right_dividend` 最近一次實際現金股利,**2010 年起每一年、兩個市場的中位數比值
都是 1.000**。2009 年以前中位數 0.74~0.89,原因是那個年代普遍發股票股利、
`ex_right_dividend.cash_dividend` 存的是含權益調整的數字(例:台積電 2006-06-20
記 4.7691,實際現金股利 2.5),不是 `stock_per_pbr` 的問題;而且方向是**低估**,
與前視相反。

結論:**存進資料庫的 `dividend_yield` 全期都是當下可知的資訊,單位是百分比
(4.50 = 4.5%)。**

### 🟡 SUSPECT — 來源端的極端值原封照抄

殖利率 > 30% 的有 twse 3,552 列、tpex 4,965 列;最極端的是全銓 8913 在 2018-10~11
的 **763.76%**。查原始列 `"8913","全銓","248.33","113.80000000","106","763.76","1.21"`
——來源自己用「每股股利 113.80 ÷ 收盤 14.90」算出來的(全銓當年有巨額減資/清算性
分配)。解析沒問題,是來源的語意特殊。任何用殖利率排序的策略必須自己設上限,
否則這類股會長期霸榜。同理股價淨值比 > 100 的有 twse 789 / tpex 1,520 列
(多為淨值極小的生技股),本益比 > 5000 的 twse 490 / tpex 718 列。

### ⚪ 附帶觀察(交給別的稽核單位)

- **cache 比 PG 新**:`var/cache/cache.duckdb` 多了 2026-07-20 的 1,968 列(tpex 889 +
  twse 1,079),PG 最新只到 2026-07-17。是 Python 直寫 DuckDB 的爬蟲跑在前面,
  方向是「cache 較新」不是「cache 過期」,但兩條寫入路徑的收斂需要有人負責。
- **`daily_quote` 缺 5 個平日**:2021-08-18、2023-06-08(tpex)、2025-08-15、
  2026-04-29、2026-05-28 這幾天 `stock_per_pbr` 有 800~1,077 檔資料,`daily_quote`
  卻一列都沒有。屬 `daily_quote` 的覆蓋缺口。
- **TPEx `daily_quote` 從 2007-07-02 才開始**,但 `stock_per_pbr` 從 2007-01-02 就有,
  所以 2007 上半年的 TPEx 估值沒有價格可對照。

## 建議修法

1. **BUG 1**:刪掉那 19 個 (market='twse', date) 的 16,447 列(PG + cache 兩邊),
   10 個真交易日重抓、9 個非交易日改寫 0-byte sentinel。守護要**兩道**:
   (a) 落檔前比對標題的民國日期是否等於請求日期(這 19 檔全部會被抓到);
   (b) 整日內容指紋去重(防 TWSE 把請求日期原樣回聲的情況——`daily_quote`
   的 2009-12-12 就是這樣躲過標題檢查的)。
2. **BUG 2**:把 `values.size match` 改成**用標頭中文欄名定位**(`本益比`/`股價淨值比`/
   `殖利率`),不要用欄數猜版型;若堅持用欄數,正確對照是
   `case 6 => (v(2), v(4), v(3))`、`case 8 => (v(4), v(5), v(2))`、`case 9 => (v(5), v(6), v(3))`,
   其餘欄數一律 `throw`(不要 `case _` 兜底)。刪掉永不觸發的 `case 7`,並改掉
   726-733 行那段與事實不符的註解。加一條回歸測試:對三個世代各一列原始文字
   斷言解析結果。
3. **BUG 3**:2-byte 空回應不得當成休市(休市 sentinel 應為 0-byte,見
   `docs/data_ops/twse_publish_times.md` 的規則);落檔前先確認該日是否為交易日
   (`src/quantlib/data_calendar.py::is_trading_day`),是交易日而拿到空回應就
   `[deferred]` 刪檔重抓。tpex 兩個缺檔直接補抓。
4. **漏欄**:把「財報年/季」接進 schema(TWSE 2017-04 起、TPEx 2025-01 起)。
   這是估值數字的 PIT 錨,現在完全沒有。
5. **去重規則**:`dataAlreadyInDB` 以 (market, 檔名) 判斷,原始檔重抓也不會覆蓋。
   改成內容雜湊比對後 upsert(與 `A-daily_trading_details` 同一條缺陷類)。
