# C-industry_taxonomy_pit — cache 一致性與缺漏

**結論:🔴 BUG。這張表叫 point-in-time,但它的產業標籤有八成不是 point-in-time
——標籤的真實年份是「我們哪一天去下載那個檔案」,不是「那筆資料的月份」。**

白話講:MOPS 的月營收彙總檔是**你按下下載那一刻才即時渲染的**,裡面的「產業別」
用的是**渲染當下的分類**,不是那個月當時的分類。我們的 2013-2019 那六年的檔案是
**2020 年 5-7 月一次補抓下來的**,所以那六年、13.7 萬列的產業標籤,全部是
**2020 年年中的分類表**被貼回歷史。表裡的 `effective_date`(資料月 +1 月 +12 天)
是我們自己算的,不是這個標籤真正被人類知道的日子。

一句話的影響:**任何「同業相對」的計算,在 2019 年以前都用到了當時還不知道的
分類。**現役 S 策略(`research/apex/strategy_s.py`)的特徵起點是 2014-10-31,
它的營收加速度要減掉「同業中位數」——同業是誰,用的就是這張表。

除了這個之外,這張表其他方面都很乾淨:月份序列零缺口、日期公式零誤差、
無重複鍵、無未來日期、抽樣逐欄比對與原始檔一致、今天(2026-07-20)有交易的
1,956 檔裡只有 3 檔查不到產業。

## 一、先講清楚:這張表在 PostgreSQL 裡不存在

`industry_taxonomy_pit` 是 **cache 專屬的衍生表**,PG 的 23 張表裡沒有它
(`psql -c "\d industry_taxonomy_pit"` → `Did not find any relation`)。

它由 `research/industry_taxonomy.py::build_industry_taxonomy_pit` 從
**cache 裡的 `operating_revenue`** 現算。兩條路徑都 import 同一支 builder,
沒有重寫、沒有漂移:

| 路徑 | 位置 | 行為 |
|---|---|---|
| 全量重建 | `research/cache_tables.py:103-105` | `CREATE TABLE ... AS` |
| 每日增量 | `research/crawl/sources/operating_revenue.py:107-118` | `DROP TABLE` + `CREATE TABLE` |
| pg-attach parity | `research/db.py:53-66` | 沒有實體表時即時建 TEMP VIEW |

所以「cache vs PG 一致性」的正確問法是:**用 PG 的 operating_revenue 重跑同一支
builder,結果和 cache 裡那張表一不一樣?** 答案:99.98% 一樣,不一樣的部分
100% 是繼承自 `operating_revenue` 已知的雙寫入者問題(見 C-operating_revenue)。

| 比對 | 結果 |
|---|---|
| cache 列數 | 452,891 |
| PG 重算列數 | 452,835 |
| cache 獨有 | 63 列(全在 202604-202606) |
| PG 獨有 | 7 列(3426 台興 / 4987 科誠 / 5236 凌陽創新 / 6806 森崴能源) |
| 共同鍵重複鍵 | 兩邊皆 0 |
| 共同鍵逐欄不同 | company_name 6、raw_industry 14、industry 14、broad_sector 10;`source_year/source_month/effective_date/is_financial/is_special_category/industry_source/taxonomy_version` **全同** |

重跑:`/private/tmp/.../parity.py` 的做法是
`ATTACH pg` → 建 `operating_revenue` view → 呼叫 `build_industry_taxonomy_pit(p)`
→ 以 `(market, company_code, source_ym)` 為鍵做 anti-join + 逐欄比對。

## 二、頭號問題:產業標籤的「已知日」= 下載日,不是資料月

### 2.1 機制(第一手證據,11 個檔 11/11 命中)

原始 CSV 的第一欄就是檔案自己的版本戳:**出表日期**。同一家公司 3687 歐買尬,
在不同資料月的檔案裡產業別完全由**出表日期**決定,和資料月毫無關係:

```
$ grep -m1 '"3687"' data/operating_revenue/tpex/2013/2013_3_c.csv  | cut -d, -f1-5
"109/06/01","102/3","3687","歐買尬","文化創意業"
$ grep -m1 '"3687"' data/operating_revenue/tpex/2013/2013_12_c.csv | cut -d, -f1-5
"109/06/09","102/12","3687","歐買尬","電子商務"
```

完整對照(11 個檔):

| 出表日期(渲染日) | 資料月 | 產業別 |
|---|---|---|
| 2020-06-01 | 2013/3、2013/4、2013/11、2014/2、2014/3、2014/4、2014/6、2014/12、2015/2 | 文化創意業 |
| 2020-06-02 | 2013/2 | **電子商務** |
| 2020-06-09 | 2013/12、2014/1、2014/5 | **電子商務** |
| 2020-07-12 | 2015/1 | **電子商務** |

歐買尬在 **2020-06-01 到 06-02 之間**被 TPEx 從文化創意業改成電子商務。凡是那天
之後渲染的檔案,不管資料月是 2013 還是 2015,一律寫新分類。這就是為什麼分類表
裡會出現「單月閃斷」(A→B→A)——**8 筆假變動、4 檔股票**,全部是渲染日的產物,
不是真的改過產業。

### 2.2 規模(全表逐列量測)

把每一列的 `effective_date` 拿去減它來源檔的出表日期,得到「這個標籤其實是幾天
之後才知道的」:

| 前視幅度 | 列數 | 佔比 |
|---|---|---|
| ≤31 天(可接受) | 79,766 | **17.6%** |
| 1 個月 ~ 1 年 | 195,851 | 43.2% |
| 1 ~ 3 年 | 81,644 | 18.0% |
| >3 年 | 95,630 | 21.1% |

逐年中位前視天數:

| 年 | 中位前視 | 年 | 中位前視 |
|---|---|---|---|
| 2001-2008 | ~361 天 | 2016 | 1,414 天 |
| 2009 | 357 天 | 2017 | 1,046 天 |
| 2010 | 1,014 天 | 2018 | 681 天 |
| 2011 | 649 天 | 2019 | 323 天 |
| 2012 | 1,062 天 | **2020-2021** | **5-8 天** ✅ |
| 2013 | **2,501 天** | 2022 | 180 天 |
| 2014 | 2,136 天 | 2023-2025 | 21-52 天 |
| 2015 | 1,771 天 | 2026 | 4 天 ✅ |

**2013-01 到 2019-12 的 137,013 列,來源檔全部渲染於 2020-05-31 ~ 2020-07-18。**
也就是說我們根本**沒有**那六年的當期產業分類——我們有的是 2020 年的快照。

### 2.3 一個具體的失真:電子類分拆早了 13 個月

我們的表在 `source_ym=200606`(effective_date 2006-07-13)開始出現半導體業 /
光電業 / 電子零組件業,之前是「電子工業」。但供給這兩個月標籤的原始檔:

```
2006_5_i.html  出表日期:96/06/09 (2007-06-09) → 只有「化學生技醫療」等舊分節
2006_6_i.html  出表日期:96/07/09 (2007-07-09) → 已有「光電業」「電子零組件業」
```

真正的分類變更發生在 **2007-06-09 與 2007-07-09 之間**,我們把它記成了 2006-07。
受影響:`source_ym 200606..200706` 之間 **8,869 列 / 714 檔**帶著提早一年的
電子子產業標籤。(對照:TWSE 官方將電子類分拆為八個子產業的生效日為 2007-07-02,
與上面兩個渲染日完全吻合——此為外部佐證,結論本身不依賴它。)

### 2.4 這條路通到錢

| 消費端 | 位置 | 用途 |
|---|---|---|
| S 策略唯一真源 | `research/apex/strategy_s.py:47` | `accel_rel` = 營收加速度 − 同業中位數 |
| 每日 live 顧問 | `research/tri/advisors.py:233` | 逐日池籍 `pool_history` |
| 池籍追溯 | `research/tri/pool_trace.py:58`、`research/tri/tests/test_pool_history.py:56` | 同上 |

`strategy_s.py:22` 的特徵起點 `DS = "2014-10-31"`。從那天到 2019-12 為止,
S 的「同業」定義全部來自 2020 年的分類表。

**該不該恐慌?** 產業重分類本來就罕見:在乾淨的當期資料區間(2021 起,2023 官方
大改版除外)每年約 12-20 檔改分類 ≈ 0.7%/年。所以 2013 年的標籤大約有 5% 上下
會是錯的——不是災難,但也不是零,而且**電子分拆那 13 個月是整片電子股一起錯**。

## 三、其他實測結果

### 3.1 沒問題的(查過,別再查一次)

| 項目 | 結果 |
|---|---|
| 月份覆蓋 | twse / tpex 各 200106..202606 共 **301 個月,零缺口** |
| `effective_date` 公式 | 452,891 列 100% 等於 `資料月1日 +1月 +12天`,全部落在每月 13 號 |
| 未來日期 / NULL | 各 **0** 列(`company_code`/`industry`/`broad_sector` 皆無 NULL) |
| 重複鍵 | `(market, company_code, source_ym)` 重複 **0** |
| 抽樣逐欄 | 3 個月 × 5 檔(201007 / 201803 / 202411)全 15 列與原始檔 產業別 一致 |
| `is_financial` | 金融保險 11,701 + 金融業 2,060;含金控那條超長括號說明也被正確歸一 |
| `is_special_category` | 管理股票 817 列 / 47 檔、存託憑證 173 列 / 1 檔(9105 泰金寶) |
| PIT 語義的下界 | effective_date 是每月 13 號,晚於 MOPS 公告截止(次月 10 號)→ **公告延遲本身沒有前視** |
| pg-attach parity | `research/db.py:53-66` 會自動即時建表,無 cache 時查詢不會炸 |

### 3.2 公司層的覆蓋洞(不是休市,是來源只抓一半)

日曆缺口不適用(月頻表),但**公司層**有洞:

| 年 | 當年有交易的 4 碼個股 | 當年完全查無月營收 |
|---|---|---|
| 2013 | 1,514 | **114** |
| 2014 | 1,557 | 107 |
| 2015 | 1,602 | 95 |
| 2016 | 1,647 | 82 |
| 2017 | 1,673 | 63 |
| 2020 | 1,752 | 10 |
| 2026 | 1,981 | 4 |

原因:我們只抓 MOPS 的 `t21sc03`(**合併**營收),沒抓 `t21sc04`(個別營收)。
2013 年 IFRS 上路後只申報個別營收的公司整片消失。實例:1704 榮化、1902 台紙、
3315 皇田、3474 華亞科、3559 全智科,`operating_revenue` 裡只有
`type='individual'` 且**止於 201212**;`grep '"1704"' data/operating_revenue/twse/2015/2015_6_c.csv` → 0 筆。

後果分兩種:
- **標籤凍結**:2015-06-30 有 **78 檔**在交易,但產業標籤停在 2013-01-13(898 天前)。
- **完全沒有標籤**:2026-07-20 有 3 檔(9103、9110、9136,外國發行人 DR,不申報
  MOPS 月營收)——佔比 0.15%,可接受。

另外 **6 碼 TDR 被硬排除**:`COMMON_CODE_PATTERN = '^[1-9][0-9]{3}$'`
(`research/industry_taxonomy.py:22`)只收 4 碼,912000 晨訊科-DR(3,727 個交易日)、
912398 友佳-DR、910069 新曄永遠沒有產業標籤。

### 3.3 小刺(不影響數值正確性)

**(a) 每日增量重算會把索引弄丟。**
`cache_tables.py:133-134` 建了 `idx_itp_code_date` 與 `idx_itp_market_code_date`,
但 `crawl/sources/operating_revenue.py:113` 用 `DROP TABLE IF EXISTS` + `CREATE TABLE`
重算,索引跟著消失且沒重建。實測現在:

```
SELECT * FROM duckdb_indexes() WHERE table_name='industry_taxonomy_pit'  → 0 rows
```

其他 21 張表索引都在。單點查詢 283 ms(全掃)。純效能,值沒錯。

**(b) `broad_sector` 有 fallback 洩漏。**
`broad_sector()` 最後一行 `return industry`,未映射的分類直接把原名當粗分類:
其他 27,401 列、綜合 49 列、水泥窯製營造 32 列。

**(c) 跨市場同日重複 9 對(18 列),其中 1 對兩邊產業不同。**
3454 晶睿在 2010-01-13 同時有 tpex(通信網路業)與 twse(光電業)兩列。
消費端 SQL(`strategy_s.py:47`、`advisors.py:233`)**不篩 market**,
`join_asof` 平手時取哪一列不確定。只有這 1 檔,影響極小。

**(d) 最近 3 個月的標籤會被改寫。**
`_REFRESH_MONTHS = 3` 每天重抓最近三個月,而 MOPS 會用**當下**的分類重新渲染。
實測 202604 有 14 檔標籤在 5 月版與 7 月版之間變了(1595 川寶 電子零組件業→半導體業、
2230 泰茂 電機機械→居家生活、3313 斐成 其他→建材營造 …)。所以同一支回測
**跑 `cache_tables.py` 之前跟之後會得到不同的池籍**。

## 四、建議修法(不動手,交主流程裁決)

1. **把「出表日期」變成第一等公民**(治本)。原始檔第一欄就是它,現在被
   `FinancialReader.readOperatingRevenue` 丟掉、PG schema 也沒有這一欄。
   加進 Slick Table + cache + builder,然後
   `effective_date = max(資料月+1月+12天, 出表日期)`。這樣標籤永遠不會早於它
   真正被知道的那天;代價是 2013-2019 的產業會變成「2020-06 以後才可用」——
   **那才是誠實的答案**。
2. **在那之前,先在表上標風險**:加一欄 `label_vintage`(= 出表日期)或至少把
   本報告的逐年前視天數表寫進 `research/industry_taxonomy.py` docstring,讓下一個
   人不會以為 2013 年的產業是 2013 年知道的。
3. **回補 t21sc04(個別營收)**:`https://mopsov.twse.com.tw/server-java/FileDownLoad`
   POST `step=9&functionName=show_file&filePath=/home/html/nas/t21/sii/(或 /otc/)&fileName=t21sc04_{roc}_{m}.csv`,
   補 2013-01 起。可填掉 2013-2017 那 63-114 檔/年的洞。
4. **索引**:`rebuild_industry_taxonomy` 改成 `DELETE + INSERT`(和 `Sink.upsert`
   一致,索引會留著),或在 `CREATE TABLE` 後補建兩個索引。
5. **代號正則**放寬到 `^\d{4,6}[0-9A-Z]?$` 以收 TDR;`broad_sector` 為
   綜合 / 水泥窯製營造 補映射或明確歸「其他」。
6. **消費端補 market 篩選**:`strategy_s.py:47` 與 `advisors.py:233` 的
   `SELECT ... FROM industry_taxonomy_pit` 加上 `market`,並在 join_asof 帶上
   `by=["market","company_code"]`。
7. **守護測試**:對 `industry_taxonomy_pit` 加一條「每檔每年產業變動次數 ≤ 1」
   與「(company_code, effective_date) 唯一」的斷言;再加一條 parity 測試鎖住
   `cache_tables.py` 與 `crawl` 兩條重算路徑輸出逐位一致。
