# C-tdcc_shareholding — cache 與 PostgreSQL 一致性 + 覆蓋缺口

**單位**：`C-tdcc_shareholding`（dim C，表 `tdcc_shareholding` = TDCC 集保股權分散表／千張大戶籌碼）
**結論**：`SUSPECT`

## 一句話

cache 這份表抄 PostgreSQL 抄得**一字不差、可以放心用**；但這張表是**週頻、來源只給當週**，
歷史一漏就補不回來——已收集的區間裡確實**漏了兩週**（2026-04-30、2026-05-29，兩天都查證是
真交易日、不是休市）。這兩週補不回來要走 TDCC 歷史查詢介面（Task #20 待辦），不是普通重抓。
今天沒有任何程式吃到這個洞。

## cache 能不能信？能——而且是全體核對過，不是抽樣

| 檢查 | 結果 |
|---|---|
| 欄位 | PG 7 欄、cache 6 欄，**只少一個自增 `id`**（研究碼零消費者），型別零降級 |
| 列數 | 813,110 == 813,110，12 個週快照，範圍 2026-04-17 .. 2026-07-17，兩邊相等 |
| 全量指紋 | 12 個資料日 ×（列數＋六欄雜湊和＋XOR）**逐格相等**，mismatch=0 |
| 抽樣逐欄 | 3 日期 × 5 檔（85 列/組）`DataFrame.equals` 全 True |
| 兩存取模式 | `cache_tables.py:60` 建表 SQL 與 `db.py:145-148` pg-attach view 投影相同 |

被丟掉的 `id` 欄在 `research/` 全文搜尋只有兩個 tdcc 消費者，一個是線上探針（用 URL 的 `id=1-5`
參數、不是欄位），一個 `taiwan_equity_etf_ranking.py` 只讀 `max(data_date)` 與人數/股數加總——
**沒有人讀 `id`**，丟掉安全。

## 缺口：兩週真的漏抓（不是休市）

這張表是**週頻**：TDCC 的 opendata 端點沒有日期參數，永遠只回「當週最新」快照，資料日期 =
每週最後一個營業日（遇假日不必然是週五）。所以缺口的單位是「週」。

用 TWSE 的實際交易日曆（含颱風假）推每一週應有的快照，已收集區間內應有 **14 個**、只捕捉到 **12 個**：

| 應有週末營業日 | 星期 | 狀態 | 說明 |
|---|---|---|---|
| 2026-04-30 | 週四 | **MISS** | 週五 05-01 勞動節休市 → 該週最後交易日=04-30（daily_quote 1,355 列，真交易日） |
| 2026-05-29 | 週五 | **MISS** | 正常週五（daily_quote 1,362 列，真交易日） |
| 2026-06-18 | 週四 | OK | 週五 06-19 端午，正確捕捉週四 |
| 2026-07-09 | 週四 | OK | 週五 07-10 颱風休市，正確捕捉週四 |
| 其餘 10 週 | 週五 | OK | |

**為什麼漏**：看原始下載檔的內容日期就清楚了——4/30 那天下載回來的還是 04-24 週的資料
（04-30 週的快照那時還沒發布），下一次下載跳到 5/10 已經是 05-08 週；同理 5/26 下載回 05-22、
下一次 6/11 跳到 06-05。**那兩週的快照確實產生過，但活在沒有跑爬蟲的空窗期、被下一週覆蓋掉了。**

- 勞動節 05-01 本身不是交易日（daily_quote 零列），正確地不存在，**不是缺口**。
- 端午（06-19）、颱風（07-10）那兩週用週四代替、都有抓到，**不是缺口**。

**影響**：今天零影響。研究碼裡沒有任何程式對 tdcc 做「本週 vs 上週」的差分；唯一會讀資料的
`taiwan_equity_etf_ranking.py` 只看最新一週。**風險在未來**：若拿 tier 15 千張大戶做週對週變化
（`shift(1)`），跨過這兩個洞的差分會被拉成兩週間距、算歪。

## 數值：全乾淨

負人數／負股數／負比例、比例 > 100、級距超出 1~17、未來日期、NULL、重複主鍵、
「有股數卻零持有人」——**九項全部 0**。合計級距（tier 17）每一檔比例都剛好 100（47,830 檔零偏離），
證明欄位沒被 cache 錯位。

**順帶修一個記載**：A-tdcc 寫的「合計人數 == sum(tier 1..16)」精確講應是 **sum(tier 1..15)**。
差異數（tier 16，帳列與實際不符的股數，通常 1~5 人）是**獨立對帳列、不計入合計**——
合計 == sum(級距 1..15) 對全部 47,830 個（日期,股票）成立。A-tdcc 那句在 2330 剛好成立，
只因 2330 的 tier 16 人數=0。消費端註記：算「總人數／總股數」用 tier 17 或 sum(1..15)，
別用 sum(1..16)（會把差異數重複計進去）。

## 小事：cache 缺唯一索引

cache 的兩個索引都不是唯一的，PG 側有 `UNIQUE (data_date, company_code, holding_tier)`。
目前實測 0 重複，唯一性全靠「每次砍表重建」維持；未來若加增量寫入就沒東西擋重複列。
建表後補一條 `CREATE UNIQUE INDEX` 即可（與 sbl 同修法）。

## 修法（給主流程，不自行下載）

1. **回補兩週**（2026-04-30、2026-05-29）：**不能用標準 opendata 端點**（只給當週）。走 TDCC 歷史
   查詢介面（集保戶股權分散表查詢 `https://www.tdcc.com.tw/portal/zh/smWeb/qryStock`，可按日期查
   歷史週）——即 CLAUDE.md 資料路線圖與 A-tdcc 已記的「歷史回補 pending（Task #20）」。
2. **防復發**：每次 crawl 後斷言「已收集區間內每個 ISO 週的 TWSE 最後交易日都出現在 tdcc
   `data_date` 集合」（現在跑紅 2 筆，回補後歸零），把週頻漏抓從無聲變可觀測。
3. **索引對齊**：cache 建表後補 `CREATE UNIQUE INDEX ON tdcc_shareholding(data_date, company_code, holding_tier)`。
