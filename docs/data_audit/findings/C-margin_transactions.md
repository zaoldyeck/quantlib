# C-margin_transactions — 融資融券表:cache 一致性與缺漏

**結論(一句話)**:cache 跟資料庫一模一樣、一位都不差,所以「複製過程有沒有走樣」
可以放心;但**資料本身有 11 個交易日整天不見、10 天存的是別天的複製品、還有 13.6 萬列
的融券限額整欄裝錯東西**——這三件全部原封傳進 cache,而且都不會自己修好。

**判定**:🔴 BUG(cache 一致性 🟢 OK,資料齊備性 🔴 BUG)

執行日:2026-07-22 · cache 檔:`var/cache/cache.duckdb`(mtime 2026-07-21 08:16)

---

## 一、能信的部分(先講好消息)

**cache 是 PostgreSQL 的忠實複本,沒有任何走樣。**這不是抽樣抽出來的,是全部比對過:

| 檢查 | 結果 |
|---|---|
| 全表列數 | PG 8,341,265 = cache 8,341,265 |
| 逐年逐市場列數 | 46 個「年 × 市場」格子,格格相等 |
| 全史逐日指紋(列數 + 兩種雜湊) | 11,081 個「市場 × 日期」,零筆不符、零筆只在一邊 |
| 抽樣逐欄(3 日期 × 5 檔) | 全部逐欄相同 |
| 欄位型別 | 文字→VARCHAR、日期→DATE、整數→INTEGER,無降級 |
| 兩種讀取模式是否一致 | `cache_tables.py` 與 `db.py` 的取數 SQL **逐字元相同** |

數值也不會爆掉:最大值是融資限額 1,387 萬張,離整數上限(21.4 億)還有兩個數量級,
不像法人買賣超那張表有溢位風險。

被丟掉的 11 個欄位(公司名、買進、賣出、現償、前日餘額、資券相抵……)在整個
`research/` 與 `docs/` 裡**沒有任何人用**,丟掉不影響任何人。

**異常值掃描也全清**:沒有重複、沒有未來日期、沒有負數、沒有空值、沒有怪代號。

## 二、不能信的部分

### 🔴 1. 11 個真的有開市的日子,整天沒有資料(約 8,618 列)

| 市場 | 日期 |
|---|---|
| twse | **2002-10-24**、2004-08-20、2007-09-06、2008-11-28、2010-08-03、2011-03-28、2011-06-23、2013-08-30、2014-05-23、2018-11-01 |
| tpex | 2023-06-09 |

**2002-10-24 是這次新抓到的**,前一輪(A 維)的稽核看不到它——因為那輪拿報價表當
日曆,而報價表只到 2004 年才開始,2002 年整段是盲區。這次改用三種方法交叉:
其他表當證人、純日曆推算、**看原始檔長什麼樣**。

第三種方法是這次的關鍵發現:交易所對「這天沒有資料」的回應,固定是
818 bytes(舊版空表)、26 bytes(新版「很抱歉,沒有符合條件的資料!」)或
0 bytes(爬蟲自己寫的休市標記);而**只有 4 bytes 的檔案是抓失敗**。全部 4-byte 的
檔案有 12 個,扣掉 2 個週六,剩下 10 個就是 twse 的缺漏日。

再拿交易所自己的資料覆核(TWSE 每日市場成交資訊,查了 6 個月):
2002-10-24 那天大盤收 **4,574.80**、跌 15.08 點——確實有開市;
反向查 2001-01-19、2001-09-12/17/18、2002-09-06、2003-09-11,交易所自己的清單裡
也沒有,證明「818/26 bytes = 休市」這條判別法可靠。

**為什麼永遠不會自己補**:爬蟲看到「檔案存在」就跳過(不管檔案是不是空的),
匯入程式看到「這個檔名已進過資料庫」也跳過。兩條路都不回頭。所以要補,得**先刪掉
那些空檔再重抓**,順序反了沒用。

### 🔴 2. 10 天存的是別天的複製品(6,068 列)

- **7 個颱風休市日**(上櫃 2012-08-02、2014-07-23、2015-07-10、2015-09-29、
  2016-07-08、2016-09-27、2016-09-28)明明沒開市卻有整天的資料,內容是前一交易日的複本。
- **上櫃 2008-08-29**(真交易日)存的是 08-28 的資料,那天的真資料遺失了。
- **上市 2003-09-12** 存的是 09-18 的資料。
- **上市 2011-03-26**(週六)存的是 **2017-12-18** 的資料——**六年後的資料**。
  用到這天的回測等於偷看未來。

這次是用 cache 自己的資料獨立重現的:把每一天的內容壓成一個指紋去撞號,
用「餘額」兩欄比對時剛好撞出 9 對,跟前一輪從原始檔查出來的 9 對完全一樣。
(用全部 4 欄比只撞到 6 對,因為限額是每天依流通股數重算的,休市日照樣會更新,
只有餘額是被整份複製的。這個細節本身就是複製行為的旁證。)

### 🔴 3. 上櫃 2007-06 ~ 2008-09 的「融券限額」整欄裝錯東西(13.6 萬列)

那 16 個月裡有 **48.51% 的資料出現「融券餘額比融券限額還大」**——物理上不可能。
同期的限額平均值從 36,375 掉到 204.6,兩年後又跳回 35,285,斷層乾乾淨淨。

決定性的一刀:全表「餘額 > 限額且限額不為 0」共 33,001 列,**100% 落在上櫃
2007 與 2008 年**,2009 年之後和上市市場全史一列都沒有。也就是說其他時候出現的
「餘額 > 限額」都是合法情形(停券但餘額還在),只有這 16 個月是真的錯。

根因在前一輪已經查清(解析程式寫死欄位順序,而那 16 個月的檔案少印了一欄),
這裡只是確認**錯誤原封傳進了 cache**——因為 cache 的取數就是一句純搬運的 SQL,
資料庫錯什麼它就錯什麼。

## 三、要注意但還不算錯的

### 🟡 cache 落後兩個交易日,而且表跟表對不齊

融資融券只到 **7/17**,但同一份 cache 裡的報價、法人買賣超、本益比已經有 **7/20**,
而「該齊備」的日子是 **7/21**。原因是融資融券沒有納入 Python 的每日直抓清單,
只能靠 Scala 爬蟲抓進資料庫、再整份重建 cache,而最後一次 Scala 爬蟲跑在 7/19 晚上。

這正是 CLAUDE.md 記載的 2026-07-15 事故形態(表間日期錯位 → 策略閘門查無資料 →
把候選靜靜砍光)。**跑一次完整更新就能拉平**。

### 🟡 repo 自己的融資融券體檢是一道死掉的守護

`research/audits/03_full_data_audit.py` 檢查的是「融資餘額 > 融資限額的 **100 倍**」
且「一天要超過 5 筆才報告」。全史 834 萬列裡,滿足前者的**只有 1 列**——這個檢查
從來沒有、也永遠不可能亮紅燈。它既看不到 84,057 列「限額 0 但仍有餘額」,
也完全沒有檢查融券那一側(上面那 33,001 列物理不可能的組合)。

那個 `* 100` 是沒有出處的魔術數字,應該換成有物理意義的兩條斷言。

### 🟡 cache 沒有唯一索引

資料庫那邊「市場 + 日期 + 股票代號」是唯一鍵,cache 這邊只有一個非唯一索引。
目前實測沒有重複,但唯一性完全靠「每次砍掉重建」這條自律撐著。

## 四、看起來像錯、其實是真的(別再查一次)

### ⚪ 限額是 0 但還有餘額(8.4 萬列),或四個欄位全是 0(26.5 萬列)

那是**停資停券**。交易所把限額設成 0,但先前已經建立的餘額還掛在那裡。
原始檔的「註記」欄會印 O(停止融資)/ X(停止融券),但 cache 沒接這一欄,
所以從 cache 看只剩一個突兀的 0。

實例:2019-09-05 的 2609 陽明,原始檔印 `"0","0","0","0","0",... ,"OX "`,
融資餘額 5,447 張、限額 0。同一天的 2330 台積電註記是空白、限額 648 萬張。

**消費端要注意**:算「融資使用率」一定要先擋掉限額 0 的列。Scala 那邊
(`Signals.scala:390`)有這道閘門;Python 這邊 `r09_finlab_harvest.py:99` 寫的是
`margin_balance / (margin_quota + 1)`,對這 8.4 萬列會算出天文數字。

### ⚪ 2017-08-11 之後「全 0」的列突然變多

不是我們解析變了,是**交易所印法變了**。在那之前,停資停券股票的限額欄照印真實
數字;從 2017-08-11 起改印 0,而且**兩個市場同一天改**。逐格對原始檔確認:
1413 宏洲在 2017-08-10 限額 25,528、2017-08-11 變 0,註記兩天都是 OX。

這對跨年代的融資使用率因子是隱形陷阱——同一檔停資股票在 2017-08-10 算得出使用率、
隔天變成除以零,會在那一天製造一個人造的截面斷層。

### ⚪ 184 列「融資餘額 > 融資限額」

限額調降(減資、融資成數調整、標的資格取消)時,先前的餘額不會被強制平倉,
所以會暫時高於新限額。占比 0.0022%,主要是 2001 年的 2410 鼎大(停資停券中)
和 2018-2019 的 00686R(反向 ETF,受益權單位隨買回縮減)。原始檔就是這樣印的。

### ⚪ 409 個「平日卻沒資料」的日子

全部是休市(春節連假、228、清明、勞動節、端午、中秋、國慶,以及 2001-07-30 桃芝
颱風、2001-09-17/18 納莉颱風)。2001~2004 那幾年沒有任何其他表可以當證人,
所以是靠原始檔的回應形態 + 交易所第一手資料判定的。
