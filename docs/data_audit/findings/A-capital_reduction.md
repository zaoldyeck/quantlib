# A-capital_reduction 解析正確性稽核

**一句話結論:這份減資資料可以信。** 原始檔每一欄都正確、完整、單位無誤地落到
`capital_reduction` 表——我不呼叫既有解析程式、自己重讀 206 個原始檔重解出 666 筆,
跟資料庫逐欄比對**零差異、零漏筆**。唯一要記在牆上的是「除權參考價」這個幾乎沒人用的
欄位有兩個語意小眉角(見下),完全不碰策略會用到的價格 / 日期 / 代號。

## 原始檔結構(跨時代抽樣解碼後)

**TWSE**(每列 12 欄,程式濾 `row.size==12 && head!="恢復買賣日期"`):
`恢復買賣日期(民國 y/m/d) | 股票代號 | 名稱 | 停止買賣前收盤價 | 恢復買賣參考價 |
漲停價 | 跌停價 | 開盤競價基準 | 除權參考價(無=「--」) | 減資原因 | 詳細資料(請點選) | 空尾欄`
→ 程式用到 col0~9,**丟棄 col10（詳細資料，只是「請點選」連結、無 per-row 資料）+ col11（尾逗號空欄）**,無資訊損失。

**TPEx**(每列 10 欄,程式濾 `row.size==10 && head 符合 \d{7}`):
`恢復買賣日期(民國 yyyMMdd) | 股票代號 | 名稱 | 最後交易日收盤價 | 減資恢復參考價 |
漲停價 | 跌停價 | 開始交易基準價 | 除權參考價(無=「0.00」) | 減資原因`
→ 10 欄全用。

**注意**：這兩種檔都是**全史快照重抓**(如 `2026_5_20.csv` 首行 `-1900年... 至 106年...`、
內容從民國100 中華電一路到今),所以**檔名日 ≠ 內容日是正常的**,不是 crawler bug。

## 怎麼查的(可重跑)

1. 讀解析程式 + schema + 設定,寫下宣告的欄位順序 / 型別 / 單位 / 濾條件。
2. 跨時代抽樣:twse+tpex 各 2020(最早)/2023(中)/2026(最近)用 `iconv -f BIG5-HKSCS` 解碼看結構。
3. **獨立 Python 重解**(`scratchpad/audit_capred.py`,不呼叫 Scala reader):自己的欄位判讀
   讀全部 206 非哨兵檔 → 666 筆,與 `psql` dump 的 666 筆**逐欄比對**。
4. 專盯:漏筆(size 濾條件是否誤殺)、欄位錯位、單位、正負、編碼、日期、漏欄。

## 結果

### 通過(可信)

- **完整 + 無錯位**:獨立重解 666 = PG 666,key 完全一致(`in_PG_only=0`、`in_INDEPENDENT_only=0`);
  所有 date-like 行 twse 全 12 欄、tpex 全 10 欄,**size 濾條件零誤殺**(`dropped_datelike=0`)。
  8 個數值/字串欄逐筆比對,除下述 4 筆版本差異外全對。
- **單位/正負**:價格 0.85~360 元/股、千分位逗號正確剝除、無負值、無異常極值。
- **編碼**:Big5-HKSCS 正確——PG 全表 0 筆 mojibake、公司代號 0 筆格式異常,中文名(南亞科、誠創…)乾淨。
- **日期**:民國→西元換算兩市場(twse `+1911` regex、tpex `MinguoChronology`)全部 666 筆與獨立換算一致。
- **漲跌停看似不是 1.1x(147/666)其實正確**:2015-06-01 台股漲跌幅從 ±7% 改 ±10% + 跳動單位捨入。
  分段驗證:2015 前 `avg(limit_up/postref)=1.0614`(±7%)、2015 後 `=1.0966`(±10%);147 偏離列 146 落 2015 前。

### 記在牆上的小眉角(非資料錯誤,下游要知道)

1. **除權參考價 NULL 語意跨市場不一致**(SUSPECT):twse 用「--」→ NULL(386 筆只 1 筆非空),
   tpex 用「0.00」→ 0.0(280 筆全 0.0)。**同樣是「這次減資沒有除權」,twse 存 NULL、tpex 存 0.0**。
   下游若對這欄做平均或篩選,得把 tpex 的 0.0 當缺值。忠實反映原始檔,但不一致。
   (reader:274 vs :287)

2. **同事件多列被唯一索引收斂成一筆,保留哪筆不確定**(SUSPECT):TWSE 全史報表把同一次減資
   列了多次,只差在**公司名版本**(2408 南亞科/南科、5906 F-台南/台南-KY、2438 英誌/翔耀)或
   **除權欄**。最尖銳的是 **3536 於 2015-03-20**:同一檔內三列、除權一列「--」兩列「13.06」,
   `distinctBy` + `.par` 平行匯入保留「先匯入」那筆 → DB 存成 `ex_right=NULL`,丟掉了 13.06 的版本。
   影響僅 1/666、且除權欄近乎沒被策略用到(twse 全表僅 1 筆非空),但這確實是「保留了較不完整的那筆」。
   **沒有任何「不同事件」被吃掉**——個股每次減資只復牌一次,所有碰撞列價格完全相同。
   (reader:291 唯一 key 去重)

3. **檔名日 ≠ 內容日(REAL,設計如此)**:全史快照源,`research/04_cross_verify` 這類檔名=內容日
   檢查會對它誤報,應列白名單。

## 建議修法(交主流程裁決,本稽核不改碼)

- 眉角 1/2 若要根治:去重時對同 key 多列採「非空優先」(coalesce 除權、取較新名),
  或把 tpex `0.00` 統一映 NULL。因除權欄近乎未用,亦可僅記錄不動。
- cross-verify 類腳本把 `capital_reduction` / `ex_right_dividend` 全史快照源列入「檔名≠內容日」白名單。

**驗證腳本**:`scratchpad/audit_capred.py`(獨立重解 + 逐欄比對,可重跑)。
