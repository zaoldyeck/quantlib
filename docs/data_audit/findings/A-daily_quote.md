# A-daily_quote — 原始檔到資料庫的解析正確性

**結論:🔴 BUG,但只是兩個小洞——每日報價的數字本身可以信。**

白話講四件事:

1. **解析是對的,而且是全史對過的,不是抽樣。**我自己另外寫了一份解析程式
   (**用欄位名稱定位、不看位置**,這樣只要 Scala 那邊欄位排錯就一定會露餡),
   把 22 年、15,151 個原始檔全部重讀一遍,再跟資料庫逐日逐欄比對:
   **10,198 個「市場 × 日期」、912 萬列、13 個數字欄的列數／空值數／欄位加總,
   一個都沒差。**另外挑 16 天做「一格一格比」也全對。
2. **有一天的原始檔下載到一半就斷了**:`2017-04-17` 的上櫃檔案在「8923 時報」
   那一列被截斷,後面 21 檔股票(8923~9962)整天沒有報價進資料庫。程式沒有任何
   「檔案有沒有下載完」的檢查,那一列被靜靜丟掉、剩下的根本沒到過。
3. **公司名稱被改過**:程式把整列的空白字元一律刪掉,所以 `元大MSCI A股`
   在資料庫裡變成 `元大MSCIA股`。數字沒事,但名稱不是原樣。
4. **上市跟上櫃的「成交量／成交金額」口徑不一樣**——這不是 bug,是兩個交易所
   自己的定義不同,但用起來會出事:**上市的量含零股、盤後定價、鉅額交易;
   上櫃的不含定價交易、而且 100% 是整張。**拿同一個成交金額門檻(例如
   日均成交額 ≥ 5,000 萬)去篩兩個市場,篩的其實不是同一件事。

另外順手記兩件「原始檔有、資料庫沒接」的事:上櫃原始檔每天都附
**發行股數**(等於免費的市值分母,387 萬個值)和次日漲跌停價,這三欄從來沒進過
schema;上櫃來源本來就沒有本益比,所以 `price_earning_ratio` 在上櫃是全 NULL。

## 二、主結論:全史逐欄核對零差異(🟢 OK)

**這是負結果,但是最重要的那個。**

方法:另寫一份獨立解析器(`_cells` 只把 `="` 正規化成 `"`,欄位一律**用標頭中文名
查位置**,不用寫死的 index),把 `data/daily_quote/{twse,tpex}/**/*.csv` 全部重讀,
每個「市場 × 日期」算出:列數、每一欄的 NULL 數、每一欄的加總;再對 PostgreSQL
下同樣的 `count / count(*) filter (where x is null) / sum(x)` 比對。

```
compared date-market pairs: 10198
only in raw (no DB rows): 0
only in DB (no raw rows): 0
MISMATCH counters: {}
```

涵蓋 **9,124,080 列 × 13 個數字欄**,零差異。這一次比對同時否定了下列所有假設:

- 欄位錯位(TPEx 2020-04-30 換版型那天前後都對得上)
- 單位錯誤(加總對得上 → 沒有 ×1000 / ÷1000)
- 正負號錯誤(`change` 的加總對得上)
- 哨兵值誤判(NULL 數逐欄對得上)

再加做 16 天「一格一格比」(含公司名稱字串),跨最早/中間/最新與兩個市場:

| 市場 | 日期 | 原始列數 = DB 列數 | 欄位不符 |
|---|---|---|---|
| twse | 2004-02-11 / 2004-12-30 / 2009-01-05 / 2015-01-05 / 2020-01-02 / 2020-03-19 / 2026-01-02 / 2026-07-17 | 711 / 721 / 742 / 911 / 1114 / 1113 / 1332 / 1371 | **0** |
| tpex | 2007-07-02 / 2012-01-02 / 2015-01-05 / 2020-04-29 / 2020-04-30 / 2024-01-02 / 2026-01-02 / 2026-07-17 | 826 / 606 / 684 / 885 / 885 / 911 / 987 / 1013 | **0** |

## 三、🔴 BUG:2017-04-17 上櫃原始檔下載到一半就斷了,21 檔股票整天不見

**證據**

```bash
# 檔案最後一行是斷的,而且沒有 TPEx 一定會有的「共N筆」結尾
iconv -f BIG5-HKSCS -t UTF-8 data/daily_quote/tpex/2017/2017_4_17.csv | tail -1
# → "8923","時報"," ----","--- ","----","----","----","0","0","0","17.05","
```

```sql
-- 當天只有 713 列,前後日都是 734~735
SELECT date, count(*) FROM daily_quote
WHERE market='tpex' AND date BETWEEN '2017-04-14' AND '2017-04-18' GROUP BY 1;
--  2017-04-14 | 734
--  2017-04-17 | 713   ← 少 21
--  2017-04-18 | 734
```

- 4,924 個有資料的 TPEx 原始檔中,只有這 1 個缺 `共N筆` 結尾;其餘 4,923 個
  檔案自報的筆數與實際解析出的列數**完全相符**(0 個不符)。
- 每日最大股票代號:TPEx 4,678 天都是 `9962`,**只有 2017-04-17 是 `8921`**;
  TWSE 5,519 天只有 `9945→9955→9958` 三段乾淨的歷史更迭,沒有斷檔。
- 列數低於前後 5 日平均 98% 的日子全市場只有 2 天,另一天 2010-07-13
  自報 `共1152筆` 與實際相符,是真的少掛牌,不是斷檔。

**丟掉的是這 21 檔**(以 2017-04-18 的清單為準):8923 時報、8924 大田、8927 北基、
8928 鉅明、8929 富堡、8930 青鋼、8931 大汽電、8932 宏大、8933 愛地雅、8934 衡平、
8935 邦泰、8936 國統、8937 合騏、8938 明安、8941 關中、8942 森鉅、9949 琉園、
9950 萬國通、9951 皇田、9960 邁達康、9962 有益。

**為什麼會靜靜過去**:`TradingReader.scala:133` 的 `filter(_.size >= 15)` 把那條
半截列丟掉(不出聲),後面 20 檔根本沒下載到,程式也沒有任何「檔案完不完整」的

**修法**:① 重抓 `https://www.tpex.org.tw/web/stock/aftertrading/otc_quotes_no1430/stk_wn1430_result.php?l=zh-tw&o=csv&se=EW&d=106/04/17`;
② 先 `DELETE FROM daily_quote WHERE market='tpex' AND date='2017-04-17'`(PG 與 cache 都要)再重讀;
③ 根因守護:TPEx 落檔前必須驗 `共N筆` 存在且等於實際列數,不符就當 `[deferred]`
刪檔重抓(TWSE 側可改驗「最後一段是備註區」或列數與前一交易日落差門檻)。

## 四、🔴 BUG:公司名稱的內部空白被抹掉

`TradingReader.scala:112` 與 `:133` 對**整列**做 `.replace(" ", "")`,目的是清掉
數字欄的空白,但公司名稱也一起被改。

```sql
SELECT market, company_code, company_name FROM daily_quote
WHERE date='2026-07-17' AND company_code IN ('00739','00890B','00928');
--  twse | 00739  | 元大MSCIA股        ← 原始檔是「元大MSCI A股」
--  tpex | 00890B | 凱基ESGBBB債15+    ← 原始檔是「凱基ESG BBB債15+」
--  tpex | 00928  | 中信上櫃ESG30      ← 原始檔是「中信上櫃ESG 30」
```

抽 240 個原始檔(兩市場各 120,隨機跨年)量到 **100 個名稱含內部空白**;光
2026-07-17 的 TWSE 一檔就有 47 列。影響:數字全部不受影響,但任何拿
`company_name` 去對外部名單(ETF 清單、公開資訊觀測站)的比對會失敗。

**修法**:只對數值欄做 `cleanCell`,名稱欄保留原樣(或只做 `trim`)。

```
filename-vs-content date mismatches: 0 []
```

(但如第五節所述,這條檢查對「伺服器回錯內容」無效。)

### 🟢 編碼:全史零解碼失敗、零列被 CSV reader 吞掉

| 檢查 | 結果 |
|---|---|
| 成交金額 ÷ 成交股數 ÷ 收盤價(應 ≈ 1) | 每個「市場 × 年」都落在 **0.9998 ~ 1.0097**(2004~2026 全部 43 組) |

### 🟢 正負號:`change` 與實際漲跌一致

|---|---|---|---|

## 七、⚪ 看起來像錯、其實是真的

### 1. 上市與上櫃的「成交量／成交金額」口徑不同(要小心)

- TWSE 原始檔備註原文:「本統計資訊含一般、**零股**、**盤後定價**、**鉅額交易**,
  不含拍賣、標購。」
- TPEx 原始檔第一行標題:「上櫃股票每日收盤行情(**不含定價**)」。

實測(成交量能被 1,000 整除的比例 = 是不是純整張):

| 日期 | twse | tpex |
|---|---|---|
| 2019-07-17 | 19.24% | **100.00%** |
| 2021-07-16 | 6.94% | **100.00%** |
| 2026-07-17 | 2.64% | **100.00%** |

上市的比例逐年下降,正是 2020-10 盤中零股交易上路後零股占比變高的軌跡。
**結論**:同一個 `trade_volume` 欄,上市是「含零股/定價/鉅額」、上櫃是「純整張、
不含定價」。任何跨市場的流動性門檻(例如 `Universe` 的日均成交額 ≥ NT$50M)在
上市這邊會偏寬。不是解析錯,但要在文件寫清楚。

### 2. 上櫃 `closing_price = 0` 共 119,680 列

全部在 2007-07-02 ~ 2011-11-11 的上櫃,且成交量都是 0——是舊版 TPEx CSV 對
「當日無成交」直接寫 `0.00`(原始檔實例:`"4801","碼斯特","0.00","--- ","0.00",…`),
2011-11-14 起才改用 `---`。解析忠實照抄。`src/quantlib/prices.py:129-130` 已有
`closing_price > 0 AND opening_price > 0` 護欄,官方定價路徑不受影響。
(此項 `C-daily_quote` 已完整記載,本單位確認來源端寫法。)

### 3. 上櫃沒有本益比、2020-04-30 前沒有最後買賣量

`price_earning_ratio` 在 tpex 全部 3,866,376 列皆為 NULL、`last_best_bid_volume`
在 2,473,019 列為 NULL——因為**來源檔就沒有這兩欄**,不是漏解析。

## 八、🟡 SUSPECT:兩個還沒炸、但機制已經在的風險

### 1. TPEx 版型判斷用 `case _` 兜底

```scala
values.size match {
  case 15 => (... transferValues(8), noneInt, transferValues(9), noneInt, noneDouble)
  case _  => (... transferValues(8), transferValues(9).map(_.toInt), transferValues(10), ...)
}
```

`case _` 的取值位置是寫死給 17 欄用的。TPEx 已經在 2020-04-30 無聲加過兩欄
(15→17);再加一次(變 16 或 18 欄)時,`filter(_.size >= 15)` 會照收,`case _`
會照跑,**買賣價量整組錯位而且不出聲**——正是 `CLAUDE.md`「Known Bug Patterns」
第一條記載的形態。TWSE 側目前是幸運:新欄若加在最後,`.init` 剛好吸收掉。

**修法**:改成 `case 15 => … case 17 => … case n => throw`(fail loud)。對照組:
Python 側 `src/quantlib/crawl/sources/daily_quote.py:65-72` 的 `_guard()` 已經在做
「標頭位置對不上就丟 `SchemaDrift`」,Scala 側該補上同樣的守護。
(附帶:該 `_guard` 的 `_TPEX_GUARD` 只寫得出 17 欄版型的位置,套到 15 欄舊檔會誤報,
但它只抓當日資料,實務上不觸發。)

### 2. 原始檔有、schema 沒接的三欄

TPEx 每日原始檔的 **`發行股數`**、`次日漲停價`、`次日跌停價` 三欄從來沒進過任何
資料表(`information_schema` 全庫查無 shares/outstanding/issued 相關欄位)。
`發行股數` 是每天現成的市值分母(3,866,376 個值、回溯到 2007),丟掉相當可惜——
現在算市值只能繞道財報或 TDCC。TWSE 側則沒有漏欄(16 欄全接)。

## 九、重跑方式

本次使用的獨立解析器與比對腳本已落地 `docs/data_audit/scripts/A-daily_quote/`
(不呼叫任何既有解析程式,欄位一律用標頭名定位;從 repo 根 `python3 <script>` 執行,
`sys.path` 內的絕對路徑請改成該目錄):

```
indep.py        獨立解析器(parse_twse / parse_tpex,依標頭中文名取欄)
fullpass.py     全史重讀 → 每日每欄 count/null/sum 指紋 + 檔名 vs 內容日期
aggcmp.py       指紋 vs PostgreSQL 同式聚合比對(主結論來源)
cmp.py          16 個日期的逐格比對(含 company_name)
trunc.py        斷檔偵測(TPEx 共N筆 結尾 / TWSE 備註區)
dropcheck.py    被欄數門檻丟掉的資料列偵測
scan_markers.py TWSE 漲跌符號值域 + CSV reader 引號規則吞列偵測
scan2.py        X/空白符號對應的漲跌價差是否為 0
scan3.py        TPEx 獨立列數統計 + 共N筆 自報值比對
namecheck.py    公司名稱空白/標頭前空行抽樣掃描
```

關鍵 SQL 一律內嵌在上文各節。
