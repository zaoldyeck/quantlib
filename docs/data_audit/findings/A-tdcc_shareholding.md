## 這份資料是幹嘛的

集保結算所每週五收盤後,把「每一檔股票被多少人、各持有多少股」按 17 個級距切開公布。
第 15 級是「持股超過 1,000 張的大戶」,第 17 級是「合計」。策略常拿「千張大戶的人數／
股數占比變化」當作籌碼面(smart money)訊號。**資料錯了,籌碼訊號就是假的。**

原始檔長這樣(6 欄,逗號分隔,UTF-8 帶 BOM,CRLF 換行):

```
資料日期,證券代號,持股分級,人數,股數,占集保庫存數比例%
20260717,2330  ,1,2398119,277360320,1.06
...
20260717,2330  ,15,1477,22019234369,84.91
20260717,2330  ,17,2941980,25932370067,100.00
```

## 怎麼查的(沒有靠猜)

另外寫了一支**不呼叫受測程式**的獨立解析器(`docs/data_audit/scripts/A-tdcc_shareholding/01_full_recon.py`,

編碼、日期、同日多檔一致性等一輪交叉驗證。

## 逐項查核結果

| 查核點 | 結果 |
|---|---|
| **逐格數值** | 813,110 格 raw vs PG,人數／股數／比例三欄**零差異** |
| **列數** | 12 個資料日逐日相等,總和 813,110 == PG 總列,**零缺零多**(沒有任何一列被 `Try` 靜默丟掉) |
| **單位** | 股數=**股**(2330 合計 259 億股,量級對)、人數=**人**、比例=**0~100 的百分比**(不是 0~1);tier 17 合計比例**每一檔都剛好 100** |
| **正負號** | 人數、股數、比例全部 ≥ 0,無負值(集保是計數與占比,本來就非負) |
| **日期** | 西元 `yyyyMMdd`(不是民國),而且取自檔案裡的「資料日期」欄、**不是檔名**——所以「檔名 4/24、內容 4/17」這種週頻落差不會把日期記錯 |
| **代號空白** | 原始檔裡 TSMC 是 `'2330  '`(右補空白到 6 字),程式 `.trim` 去掉了;PG 裡**零個代號帶空白**,對 `daily_quote` 的 join 不會斷 |
| **編碼／BOM** | UTF-8 BOM 只在標頭那行,標頭被 `\d{8}` 正則丟掉,所以 BOM 不會汙染任何資料格;入庫欄位只有代號和數字,無中文走樣 |
| **欄位錯位** | 用內部恆等式驗:2330 的「合計人數 2,941,980」== 各級距人數加總,若人數／股數欄被互換這條式子一定破 → 沒破 |
| **同日多檔** | 同一個資料日常常對到好幾個下載檔(如 20260703 對到 6 個檔),用 MD5 比對發現**位元組完全相同**,不會互相打架 |
| **cache** | DuckDB 快照 813,110 列／12 日,與 PG 完全一致 |

## 唯一一個要記的小毛病(不影響現有資料)

**平行匯入的去重閘查錯了對象。**

程式在迴圈開始前先把「DB 裡已有哪些資料日」拍成一張快照(`:769-770`),然後**平行**
跑每個檔(`:781` 的 `.par`),每個檔都拿這張**開跑前的舊快照**來判斷要不要插(`:789`)。
它另外建了一個「本次已插入哪些日」的集合 `insertedDates`(`:779`),但這個集合**從頭到尾
沒被拿來檢查**——只在插完之後 add,等於白建。

後果:**如果整庫清空重來一次**,同一個「新」資料日若同時存在好幾個檔,這幾個平行工作
會**同時**通過閘、同時去插同一天的 6.8 萬列,最後只靠資料庫的唯一索引把後到的擋掉
(輸的那幾個工作會在插入時撞 unique violation 而拋例外)。

**為什麼現在沒事:**
1. 平常是**每天增量匯入**,一個資料日在它第一次出現那天只有一個檔,閘一次就過,不會撞。
2. 就算清空重來,同一資料日的多份檔**位元組完全相同**(已用 MD5 證實),誰先搶到都寫出
   一樣的值;而且唯一索引讓批次插入是「整批成功或整批失敗」,不會插進半條——這也是為什麼
   實測列數逐日精準相等、沒有半列殘留。

**真正的殘存風險**(所以還是要記):清空重建會噴一串 unique-violation 例外(雜訊,還可能
讓整趟 `read` 顯示成失敗);而且萬一哪天 TDCC 在**同一個資料日**底下改了內容(週中更正),
存進去的會變成「平行競態誰先搶到算誰」的非決定性結果。

**建議修法**(程式健壯性,現有資料不用重匯):把 `:789` 的閘改成
`if (!existingDates.contains(dataDate) && insertedDates.add(dataDate))`——利用
`newKeySet.add` 回傳「是否首次加入」做一步到位的 test-and-set,天然免除同一趟裡同日多檔
的競態;並把插入後那次多餘的 `insertedDates.add` 拿掉。
