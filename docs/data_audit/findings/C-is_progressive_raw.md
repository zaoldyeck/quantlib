# C-is_progressive_raw — cache 一致性與缺漏稽核

- 稽核單位:`C-is_progressive_raw`(維度 C:一致性與缺漏)
- 對象:DuckDB cache 表 `is_progressive_raw`(來源 PG `concise_income_statement_progressive`,過濾 `market IN ('twse','tpex') AND type='consolidated'`)
- 日期:2026-07-22
- 結論:🔴 **BUG**

## 一句話結論

**搬運沒問題,來源有洞。** cache 與 PostgreSQL 逐季逐列完全相同,這一段可以完全信任;
不能信的是「某幾季的公司到齊了沒」——季報檔案在申報期限前被抓一次就永久凍結,
2023Q2、2024Q3、2025Q2、2026Q1 整批缺料,缺的清一色是金控/銀行/保險/KY 股。
而且缺一季不只是空一格:因為財報是累計數,少了 Q2 會讓 Q3 被算成 Q2+Q3 的合併值——
**看起來合理的錯誤數字,比空值危險得多。**

## 資料能不能信(白話版)

| 問題 | 能不能信 |
|---|---|
| cache 裡的數字跟資料庫一樣嗎? | ✅ 完全一樣。204 萬列、86 季,連整列內容的雜湊值都對得上,一筆不差。 |
| 每一季的公司都到齊了嗎? | ❌ 四個季度整批缺料,最嚴重的 2026Q1 只有 539 家(應該約 1,900 家)。 |
| 缺料只是少幾家,還是會算錯? | ❌ 會算錯。缺的那一季會讓下一季的單季數字灌水到兩倍。 |
| 2013 年以前的資料能用嗎? | ⚠️ 部分不能。營業利益整段是空的;而且每年約一成可交易公司根本不在表內。 |
| 2006-2007 的 Q1/Q3 幾乎空白,是漏抓嗎? | ⚪ 不是,當年只有半年報和年報要編合併報表。 |

## 🔴 BUG 1:季報檔案在申報期限前抓一次就永久凍結

### 現象

| 季度 | 缺料家數 | 其中當年有在交易 | 該批當年成交值 |
|---|---|---|---|
| 2023Q2 | 149 | **121** | NT$6.28 兆 |
| 2024Q3 | 11 | **10** | NT$1.93 兆 |
| 2025Q2 | 128 | **127** | NT$6.68 兆 |
| 2026Q1 | 約 1,360 | — | 整季只有 539 家 |

缺的名單(2025Q2 節錄):2801 彰銀、2880 華南金、2881 富邦金、2882 國泰金、
2885 元大金、2886 兆豐金、2891 中信金、2892 第一金、1590 亞德客-KY、3673 TPK-KY……
全是**申報期限比一般公司晚**的金控/銀行/保險/KY 股。

| 季度 | 原始檔 mtime | 金控家數 | 銀行家數 | 齊備? |
|---|---|---|---|---|
| 2023Q1 | 2026-04-23 | 13 | 10 | ✅ |
| 2023Q2 | **2023-08-20** | 子表不存在 | 3 | ❌ |
| 2023Q3 | 2024-03-12 | 14 | 11 | ✅ |
| 2024Q1 | **2024-05-16** | **1**(僅玉山金) | 11 | ❌ |
| 2024Q3 | **2024-11-22** | **4** | 11 | ❌ |

| 2025Q3 | 2025-12-10 | 13 | 10 | ✅ |
| 2025Q4 | 2026-04-01 | 13 | 10 | ✅ |

**規律非常乾淨**:下載日距季末 ≥ 約 70 天者一律完整;落在申報月當月者一律殘缺。
這排除了「法規上本來就不用申報」的可能——同一個 2023Q1,2026 年重抓就有 13 家金控。

補充證據:
- 子表數量與大小:twse 各季常態 6 個子檔約 285-320 KB;
  2023Q2 只有 4 檔 262 KB、2025Q2 只有 5 檔 281 KB、2026Q1 只有 2 檔 91 KB。
- 2026Q1 原始 CSV 的「出表日期」欄位是 `115/05/10`(2026-05-10),早於 5/15 申報期限;
  twse 310 列 + tpex 227 列 + 兩個 `_c_0` 各 1 列 = **539**,與 cache 的 539 家完全吻合
  → DB 忠實反映了殘缺的原始檔,不是匯入丟失。

### 根因(程式碼)

`src/main/scala/Task.scala:581-603` `pullQuarterlyFiles`:

```scala
val existFiles = detail.getTuplesOfExistFiles
...
val excludeYearToQuarter = LocalDate.now.getMonthValue match {
  case m if m < 3  => (thisYear - 1, 4) +: (1 to 4).map(q => (thisYear, q))
  case m if m < 5  => (1 to 4).map(q => (thisYear, q))
  case m if m < 8  => (2 to 4).map(q => (thisYear, q))   // 5/1 起就抓 Q1(期限 5/15)
  case m if m < 11 => (3 to 4).map(q => (thisYear, q))   // 8/1 起就抓 Q2(期限 8/14、金融/KY 8/31)
  case _           => Seq((thisYear, 4))                 // 11/1 起就抓 Q3(期限 11/14)
}
val future = yearToQuarter.diff(excludeYearToQuarter).filterNot(existFiles).mapInSeries {...}
```

兩個機制疊加即成災:
1. **月份閘門一律早於申報期限**——每季的閘門都在申報月 1 號就打開。
2. **`filterNot(existFiles)` 永不重抓**——只要該季已有檔案就跳過,殘檔就此凍結。

首次 `Main update` 只要落在該月,就把一份不完整的快照鎖成永久事實。

### 重現

print(con.sql('''SELECT year, quarter, COUNT(DISTINCT company_code) codes

# 2023: 1797 / 1672 / 1816 / 1951

### 修法

1. **補抓清單**:twse+tpex 的 2023Q2、2024Q1、2024Q3、2025Q2、2026Q1
   (檔名 `<year>_<quarter>_a_c_<idx>.csv`,端點即 `application.conf` 既有季報彙總來源;
   需先刪舊檔才會重抓,因為 `filterNot(existFiles)` 擋著)。
2. **合併語意而非覆蓋**——見 BUG 3,直接覆蓋會製造存活者偏誤。
3. **根因守護**:把「已有檔案就跳過」改成「該季公司家數尚未穩定就重抓」,
   判準取自資料本身(連續兩次抓取家數相同才封存),**不要再塞一個猜出來的天數常數**;
   另加回歸測試鎖死「同一年任兩季的可交易公司家數差異上限」。

## 🔴 BUG 2:缺一季不是留下空值,是把下一季算成兩季合併

財報是**年度累計數**,`research/strat_lab/raw_quarterly.py` 用「本季累計 − 上季累計」
還原單季。某季整列不存在時,polars 的 `shift(1)` 直接取到再上一季,於是下一季變成兩季相加。

### 證據:1590 亞德客-KY(2025Q2 缺料)

取自 `research/raw_quarterly.parquet`:

| 年季 | rev_q(單季營收) | ni_q(單季淨利) | rev_ttm |
|---|---|---|---|
| 2024Q1 | 7,205,592 | 1,829,164 | 37,032,944 |
| 2024Q2 | 8,427,935 | 2,260,918 | 38,555,126 |
| 2024Q3 | 7,380,792 | 1,732,258 | 30,370,109 |
| 2024Q4 | 7,645,814 | 1,800,938 | 30,660,133 |
| 2025Q1 | 8,096,083 | 1,935,652 | 31,550,624 |
| **2025Q3** | **17,198,723** ⚠️ | **4,136,403** ⚠️ | **40,321,412** ⚠️ |
| 2025Q4 | 9,035,139 | 2,327,777 | 41,975,759 |

- 2025Q3 單季營收是正常值的 **2.1 倍(+133%)**。
- `rev_ttm` 40,321,412 = 7,380,792+7,645,814+8,096,083+17,198,723,
  實際涵蓋 **五個季度**;對照 2024 全年約 30,660,133,**高估約 22%**。
- 2881 富邦金 2025Q3 `ni_q` = 49,920,909 同屬 Q2+Q3 合併值
  (2025Q1 為 41,102,222、Q4 為 29,958,655)。

**這比 NULL 危險**:所有下游因子(單季營收、單季淨利、營益率、TTM、YoY)照算不誤,
產出的是「看起來完全合理」的錯誤數字。

### 影響範圍(年內內部空洞 × 可交易性交叉)

| 年 | 缺的季 | 家數 | 當年可交易 |
|---|---|---|---|
| 2019 | Q3 | 1 | 0(興櫃) |
| 2020 | Q3 | 19 | 0(興櫃) |
| 2023 | Q2 | 149 | **121** |
| 2023 | Q3 | 4 | 2 |
| 2024 | Q2 | 1 | 0(興櫃) |
| 2024 | Q3 | 11 | **10** |
| 2025 | Q2 | 128 | **127** |

### 修法

BUG 1 補抓後這條大半自動消失。但仍應在 `raw_quarterly.py` 的 lag-diff 加防呆:
還原單季前檢查 `quarter` 與 shift 出來的 quarter 是否恰好差 1(目前只 `over(company_code, year)`
排序、**不驗連續性**),不連續就輸出 null。這樣未來再缺季,拿到的是誠實的空值而非灌水數字。

## 🔴 BUG 3:重抓舊季會抹掉已下市公司(存活者偏誤)

2023Q1 與 2023Q4 的原始檔於 **2026-04-23** 被重新下載,TWSE 彙總報表只回傳
「當下仍在市」的公司,於是 2023 年當時確實存在的 16 家從 Q1/Q4 消失,
但 Q3(2024-03-12 下載)還留著。

2023 逐季 distinct code:Q1 **1797** / Q3 **1816** / Q4 **1951**。
「Q3 有、Q1 沒有」的 22 檔,以 `daily_quote` 最後交易日檢核:

| 2443 | 2024-04-03 | | 6288 | 2025-08-04 |
| 2358 | 2024-04-03 | | 4945 | 2025-09-01 |
| 5383 | 2024-05-22 | | 2809 京城銀 | 2025-09-17 |
| 1701 中化 | 2024-08-20 | | 6747 | 2025-11-26 |
| 6514 | 2024-10-08 | | 3454 | 2026-03-18 |

**16 檔全在 2026-04-23 重抓日之前下市**。其餘 4 檔(4588/6645/6933/6937)

記警告。守護:測試斷言「同一 (year,quarter) 的公司集合只能單調成長」。

## ⚪ REAL:看起來像錯,查證後是真的

1. **2006-2007 的 Q1/Q3 幾乎空白**(每季 21-22 家)——當年只有半年報與年報需編合併報表,
   2008Q1 起立刻回到 1,041 家。原始檔即如此:`2006_1_b_c_*.csv`(合併)三檔合計 3,750 bytes,
   同季 `2006_1_b_i_0.csv`(個體)165,387 bytes,`2006_2_b_c_6.csv`(半年報合併)74,825 bytes。

2. **2014-2018 每年 22-91 家只有 Q2/Q4**——興櫃/未上市公司,只需申報半年報與年報。
   2015 年 89 家對 `daily_quote` 交叉比對:**全部 0 個交易日**。
   家數 2014 91 → 2018 22 → 2019 起 0,與興櫃申報義務逐步加嚴一致。

3. **2832 台產 2011 年 EPS 顯示 237,919 / 351,946 / 657,460 / 700,633 元**——
   TWSE **原始 CSV 本身**就這樣(`2011_4_b_c_2.csv`:營業收入=營業成本=3405881、
   基本每股盈餘=少數股權損益=700633、稀釋每股盈餘=1.93),我們的 reader 忠實搬運。
   全表掃描此錯位特徵只命中這一家一年 4 個公司季,遠在現役策略回測窗(2018+)之外。
   建議加值域守護:基本每股盈餘絕對值 > 200 元即告警(全表僅 14 筆越界)。

4. **6901 鑽石投資營業收入長期為負**——投資公司收入認列評價損益,虧損時整季收入即為負,
   與金融業月營收為負同一類。各科目內部一致(收入−成本=毛利、毛利−費用+業外=稅前),
   符合投資損失帳務結構而非欄位錯位。全表負營收僅 25 筆。

## 🟡 SUSPECT

1. **`type='consolidated'` 的時間覆蓋代價沒寫在文件裡**。
   2008-2012 每年有 112-120 檔「有在交易但只編個體財報」的公司完全不在表內
   (約占當年可交易池 10%,當年成交值 NT$580-1,003 億);1989-2005 整段空白。
   拿這張表跑 2012 年以前的回測,選股池會靜靜少掉一成,而且少掉的是
   「沒有子公司的單體企業」這種**有系統性的一群**。
   → 建議把限制寫進 `cache_tables.py` 註解與 CLAUDE.md 資料契約:
   「`is_progressive_raw` 有效起點是 2008Q1」,並把 `raw_quarterly.py --start` 預設
   從 2005-01-01 改為 2008-01-01。

2. **營業收入累計數每年 1-10 筆出現「本季累計 < 上季累計」**。
   逐年:2008 2、2011 5、2012 10、2015 4、2021 5、2025 4……
   近期樣本:3708 上緯投控 2025Q3 累計 752,966 < Q2 3,942,077;
   2601 益航 2025Q4 1,489,351 < Q3 3,147,023;2867 三商壽 2025Q2 < Q1;
   3664 安瑞-KY 2025Q4 < Q3。
   多半是處分子公司後追溯重分類為停業單位這類真實會計事件,但因下游用累計相減,
   這些筆會產生**負的單季營收**,再算營益率就會翻號。
   數量級小(每年 ≤10 筆、佔比 <0.3%),尚未逐案追到財報附註,故列 SUSPECT。
