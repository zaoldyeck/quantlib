# C-capital_reduction — cache 一致性與缺漏

**單位**:`C-capital_reduction`(維度 C:DuckDB cache vs PostgreSQL 一致性、時間覆蓋)
**日期**:2026-07-22
**判定**:🔴 **BUG**

---

## 一句話結論

**表本身乾淨得挑不出毛病(cache 和 PG 一個位元都沒差、和原始 CSV 也一個位元都沒差),
但這份對的資料在兩個地方被浪費掉:一是還原價程式把 15 筆「大比例減資」當成髒資料丟掉,
害得台股報酬序列裡憑空多出 +365%~+3626% 的假暴漲(最誇張的是南亞科 2014-09-09 的
+832%,而且它是成交量大到會被策略選進去的股票);二是這張表 2011 年以前根本沒抓,
2004~2011 的回測報酬同樣是假的。**

另外查到一件已經發生過、而且還會再發生的事:**證交所這支端點在 2024-05 ~ 2025-12 期間
對「明明有事件」的區間持續回空,27 筆減資當時全部漏抓,是 2026-05-20 端點自己出包吐出
一份全量傾印才意外補回來的。** 爬蟲對「回空」沒有任何守護。

## 四、🔴 BUG 1:還原價程式把 15 筆真減資當髒資料丟掉,做出假暴漲

### 白話

台股「彌補虧損」型的減資可以減到只剩 2.5%(1 股換 0.025 股),減完參考價會是原價的 40 倍。
`research/prices.py` 有一條護欄寫著「因子必須介於 0.05 和 5 之間,超出就丟掉」——
這條線把 15 筆真事件全丟了。丟掉之後,備援的「分割偵測」又因為「這天有減資紀錄」而
故意不動作,結果就是**價格從 8.10 跳到 75.50 完全沒有還原**,報酬序列直接記一筆 +832%。

## 六、🔴 BUG 3:證交所端點會對「有資料的區間」回空,爬蟲毫無守護

### 白話

`Task.pullCapitalReduction` 的邏輯是「從既有檔案最新那天 +1 抓到昨天」,抓完存檔;
下次再從新檔名 +1 開始。所以**一旦某次抓到空的、檔案照存,那段區間就永遠不會回頭再抓**。
2024-05 到 2025-12 就發生了:證交所連續回空,27 筆減資全部漏掉。它們今天之所以在,
純粹是因為 2026-05-20 那天端點自己出包,吐了一份從 2011 年到 2026-01 的全量傾印。

### 證據

逐檔還原「請求區間 vs 回應區間」(`05_requested_vs_returned_window.py`;非空回應的
標題會誠實回報區間,實測前期 100% 吻合請求):

```
2024_5_15.csv   req 2024-03-12~2024-05-15   ret 2024-03-12~2024-05-15   1 列
2024_5_16.csv   req 2024-05-16~2024-05-16   EMPTY(2 bytes)
2024_6_18.csv   req 2024-05-17~2024-06-18   EMPTY
...
2025_6_14.csv   req 2024-06-25~2025-06-14   EMPTY   ← 這段裡真的有 13 筆減資
2025_9_27.csv   req 2025-09-08~2025-09-27   EMPTY   ← 2025-09-15 有 2 筆
2025_10_7.csv   req 2025-10-03~2025-10-07   EMPTY   ← 2025-10-07 有 5 筆
2025_12_9.csv   req 2025-10-24~2025-12-09   EMPTY   ← 11/03、11/24×2、12/08 有 4 筆
2026_5_20.csv   req 2026-05-20~2026-05-20   → 37,250 bytes / 390 列 / 2011-01-25~2026-01-12
                                              標題「-1900年12月18日 至 106年12月18日」(亂碼)
```

`04_provenance.py` 量化:**27 筆 twse 減資(2024-07-29 ~ 2025-12-22)的第一次抓到來源
只有 2026_5_20.csv 這一個檔**,首抓延遲 149 ~ 660 天。逐年增量抓到率:

```
twse 2020 23/28  2021 27/27  2022 31/31  2023 26/26  2024 3/15  2025 2/17  2026 2/2
tpex 每年幾乎全中(2020 起 18/25、之後 21/21、20/20、15/15、10/10、10/10、4/4)
```

**端點行為現況(2026-07-22 實測,單次唯讀探測)**:

```

strDate=20240101&endDate=20260722  → 標題「115年06月29日 至 115年07月22日」,1 列
strDate=20241101&endDate=20241130  → 2 bytes(空)   ← 這個月明明有 3 筆減資
strDate=20240101&endDate=20241231  → 2 bytes(空)
strDate=20060101&endDate=20061231  → 2 bytes(空)
新平台 /rwd/zh/reduction/TWTAUU    → 0 bytes(路徑不存在)
```

→ **`strDate` 已被伺服器完全忽略**,它只回「伺服器端某個錨點 → endDate」;
endDate 只要落在過去就回空。TPEx 那支 (`bulletin/revivt`) 則完全正常,142 個檔的區間
合併起來是一整段 2013-01-02 ~ 2026-07-18,沒有洞。

### 建議修法

1. **爬蟲加 fail-loud 守護**:TWSE 這支不能再信任「回空 = 沒事件」。改成
   「回應覆蓋不到請求區間(標題起點晚於請求起點,或整份回空)就不寫檔、不推進
   `existFiles.max`」,並在 log 標記待補。
2. **改用『endDate = 今天』的滾動全量**:實測端點會回「錨點 → endDate」,把 endDate
   固定成今天就能拿到目前伺服器願意給的最大範圍,再交給 DB 去重(reader 本來就有
   `filterNot(dataAlreadyInDB)`)。這比逐段增量安全得多。
3. **加對帳守護**:每月比對「股數驟減事件」與 capital_reduction,差額超過門檻就告警
   (`10_shares_outstanding_probe.py` 已經是現成的偵測器)。

## 八、🟡 SUSPECT

### 1. 雙寫入路徑:跑 `cache_tables.py` 會把 1459 那列抹掉

`research/cache_tables.py:22-23` 先 `os.remove(DB_PATH)` 再從 PG 全量重建;
`research/crawl/sources/capital_reduction.py::refresh` 則直寫 cache。兩條路徑互相覆蓋,
與 `C-stock_per_pbr` / `C-operating_revenue` 記錄的是同一個結構病。
本表影響只有 1 列(2026-08-03 1459),但它同時是 cache 裡**唯一的未來日期列**——
消費端若沒有 `date <= 決策日` 的過濾就會前視。已盤點:`prices.py:154` 有
`date <= '{end}'`、`audits/01_find_spikes.py` 與 `02_anomaly_scan.py` 都按事件日 join,
目前**沒有實際前視發生**。

### 2. Scala reader 碰到 `-` 價格欄會炸

`src/main/scala/reader/TradingReader.scala:273`
```scala
values(3).toDouble, values(4).toDouble, values(5).toDouble, ...
```
`"-".toDouble` 會丟 `NumberFormatException`。這樣的列**現在就在證交所線上**
(`"115/08/03","1459","聯發","-","-","-","-","-","--","退還股款","請點選",`)。
目前碰不到是因為 Scala 端 `endDate = 昨天`、端點會濾掉未來列;但 2026-05-20 那種
端點異常(吐出跟請求無關的全量)一旦包含未定價列,`Main update` 會整批失敗。
Python 爬蟲的 `_price()` 有 try/except,Scala 沒有。

### 3.(跨單位)`data_calendar.is_trading_day` 認不得週六補行交易日

`research/data_calendar.py:50` 一行 `if d.weekday() >= 5: return False` 就把所有週六判死。
但 2013-09-14、2016-09-10、2017-09-30、2018-12-22 這 4 個週六是真的有開市
(872~1,072 檔上市報價)。CLAUDE.md 已把「週六補行交易日」列為真實現象,
這個日曆卻否認它。影響:`latest_complete_trading_day()` 在補行交易日當週會退回週五。
不是本表的問題,但證據是在這裡查出來的,記一筆。
