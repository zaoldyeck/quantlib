# A-taifex — TAIFEX 期貨每日行情:原始檔→DB 解析正確性

**結論(白話):** 你平常在用的「一般指數期貨」——大台 TX、小台 MTX、電子 TE、金融 TF
那些單式契約——每一欄都正確、完整、單位無誤地進了資料庫,可以放心信任。**但有一個
真的 bug:價差(組合式)契約的「價格」被無聲丟掉了。** 價差契約的報價本來就是兩個月份
的價差,天生會是負數或 0,而解析程式有一條「價格一定 >0」的過濾,把這些負/零價全部
清成空值。結果:資料庫裡 72 萬筆價差契約,有 57 萬筆查不到收盤價,其中 28.7 萬筆明明
有成交量、卻沒有成交價。單式契約完全不受影響(506 萬筆只有 31 筆邊角空值)。

**Verdict: BUG**(範圍侷限在價差/組合式契約;一般單式期貨零問題)

## 資料長相(先講清楚在測什麼)

- 落地表 `taifex_futures_daily`,PG 現有 **5,780,185** 列,日期 **1998-07-21 ~ 2026-05-21**,
  751 個契約代碼,兩個交易時段(一般 552 萬、盤後 26 萬)。
- 原始檔在 `data/taifex/futures_daily/`:1998-2025 每年一個 `YYYY_fut.csv`,2026 改成
  月檔 `2026/2026_{3,4,5}.csv`。編碼 Big5-HKSCS。
- 欄位格式**隨年代長大**(這正是無聲錯位最愛躲的地方),reader 用防禦式 `if(row.size>N)`
  守衛 + 固定索引處理,實測各世代都對齊:

  | 年代 | 欄數 | 新增欄 |
  |---|---|---|
  | 1998-2014 | 16 | — |
  | 2015-2016 | 17 | 是否因訊息面暫停交易 |
  | 2017 | 18(資料列) | 交易時段(盤後上線) |
  | 2018-2026 | 20(資料列) | 價差對單式委託成交量 + 一個尾逗號空欄 |

  > 2018+ 資料列比表頭(19 欄)多一欄,是**尾逗號**產生的 idx19 空欄——實測 30 檔
  > idx19 恆空,reader 正確忽略,零資訊損失。

## BUG:價差契約價格被 `>0` 過濾清空

**根因** `TradingReader.scala:79-80`
```scala
private def taifexOptPrice(value: String): Option[Double] =
  taifexOptDouble(value).filter(_ > 0.0)
```
`open/high/low/close/settlement_price/best_bid/best_ask/historical_high/historical_low`
九個價格欄全走這個函式。它假設「價格一定為正」,但**價差契約(契約代碼 CBF/CCF/CAF…、
到期月份含斜線如 `201301/201302`)的報價是月份間價差,合法為負或 0**,於是被整批清成 None。

**鐵證(獨立解析 vs PG 同一列):**
`2013_fut.csv` → CBF 201301/201302(2013/1/2 一般)
原始:open −0.03 / high −0.03 / low −0.05 / close −0.05 / bid −0.06 / ask −0.04 /
histH −0.02 / histL −0.05,成交量 6。
PG:`open/high/low/close/best_bid/best_ask/historical_high/historical_low` **全 NULL**,
只有 `volume=6` 存活。

**全庫規模(psql):**
```
月份含 '/' 的價差契約         723,826 列
  其中 close IS NULL         573,097 列
  其中 volume>0 且 close NULL 287,394 列   ← 確實成交、卻查無成交價
單式契約(月份無 '/')       5,056,359 列
  其中 volume>0 且 close NULL     31 列    ← 0.0006%,盤後邊角,可忽略
```
獨立掃 5 檔(2013/2018/2024/2025/2026-05):價格欄負值 624,751 個、零值 82,706 個;
負收盤 67,555 個**全部**是價差契約,單式=0。病灶專屬價差契約,證據一致。

**修法(寫在 JSON fix 欄):** 價格欄改用 `taifexOptDouble`(允許負/零,只在空/`-`/`--` 回 None),
移除 `>0` 過濾。原意(濾未成交外月佔位)其實原始檔已用 `-` 表達、`taifexOptDouble` 本就轉
None,故 `>0` 對單式多餘、對價差有害。修後重跑 read 回填價差價格,並對單式契約做
before/after 逐位 diff 確認零漂移。

**這個 bug 有多要命?** 取決於策略用不用價差契約。這個 repo 的期貨/策略用的是**單式**
指數期貨(TX 等),那部分 100% 乾淨。價差契約目前若被任何研究讀到,價格會是一片 NULL——
不是「值錯」而是「整欄消失」,屬無聲資料損失,照潔癖標準必須清償。

## 查了沒問題的部分(負結果落盤,別再查一遍)

1. **單式契約 + 所有非價格欄逐列值保真**:獨立解析(csv+big5hkscs,不呼叫受測 reader)
   7 列跨世代(1998/2014/2018/2024/2026、一般+盤後)逐欄比對 PG **全相符**——含史上
   第一筆 1998/7/21 TX(change/bid/ask 原始即空→NULL 相符)、負漲跌(−15/−72/−44/+1689)、
   盤後 settle/OI 原始 `-`→NULL 相符。
2. **聚合精確對帳**:原始「價差成交量(idx18)非空」678,153 **==** PG spread_single_volume
   非 NULL 678,153;原始「暫停交易旗標(idx16)非空」364 **==** PG trading_halt 非 NULL 364。
   兩個精確命中,證明這兩欄零漏零錯。
3. **零漏列**:有效日期卻 <16 欄的列 = 0(唯一 <16 欄者是各檔尾端 1 個 `\x1a` DOS EOF 標記,
   共 8 個,非資料)。全域 distinct 鍵 5,780,185 **==** PG rowcount,delta 0,max date 一致。
4. **編碼**:30 檔 Big5-HKSCS 全數乾淨解碼,一般/盤後 中文正確;`QuantlibCSVReader` 的
   `.replace("=","")` 對本檔為 no-op(期貨日檔無 `="..."` 包裹)。
5. **日期**:西元 `yyyy/M/d`(非民國,無 +1911 陷阱),lenient formatter 同吃 `2014/1/2`
   與 `2018/01/02`。
6. **dedup 安全**:578 萬鍵僅 1 筆碰撞(2013 MTX 價差契約在原始檔本身重複、其中一列
   historical_low=−9 髒值),依完整度收斂 1 列、符合 unique index,屬原始源異常非解析錯。

## 標註(非 bug,供下游/後續 dim 參考)

- **change_pct 是「百分比數值」不是分數**:原始 `-0.40%` 存成 `-0.40`(不是 −0.004)。
  忠實反映原始,但算報酬時要 /100。
- **Setting.validate minHeaderColumns=18**:1998-2016 歷史年檔只有 16-17 欄表頭,既有檔
  已正確入庫,但未來若重新下載這些靜態年檔會被下載驗證擋下(下載路徑 caveat)。
