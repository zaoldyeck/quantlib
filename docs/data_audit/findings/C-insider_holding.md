# C-insider_holding — cache 表 `insider_holding` 一致性與缺漏

**一句話結論:cache 和 PostgreSQL 完全一樣、一個 bit 都沒差,能不能信的問題全出在資料本身——一個會復發的解析 bug 讓 3 筆「轉讓股數」變成天文數字,而且這張表實質上只有 2026 年 3 月底之後約 150 天可用,前面 19 年整段空白。**

- **稽核維度**：C(DuckDB cache 與 PostgreSQL 的一致性 + 時間序列缺漏)
- **判定**：BUG
- **cache 檔**：`var/cache/cache.duckdb`(建於 2026-07-21)
- **重現腳本**：`docs/data_audit/scripts/C-insider_holding/audit.py`
  （`PYTHONPATH=<repo> uv run --project . python docs/data_audit/scripts/C-insider_holding/audit.py`)

## 先講 cache 一致性:完美,沒有任何問題

這是本維度的核心問題,答案很乾脆:

| 檢查 | 結果 |
|---|---|
| 欄位 schema | cache 15 欄 = PG 16 欄減掉 `id`(自增鍵,本就不該進 cache);名稱/順序/型別全對,無降級 |
| 總筆數 | cache 771 = PG 771 |
| 逐年筆數 | 2007=67、2026=704,兩邊相同 |
| **全表逐欄值比對**(非抽樣)| 771 列 × 15 欄雙向 `EXCEPT`:cache 獨有 0 筆、PG 獨有 0 筆 |
| reader 有沒有漏列 | 自寫獨立解析器數 156 個原始 HTML 檔 = **771 列 = PG = cache**,一列都沒漏 |

因為表只有 771 列,我沒有用「抽 3 天 × 5 檔」的抽樣,而是**整表每一列每一欄都比對**,這比抽樣強得多。結論:**cache 忠實鏡射 PG,沒有 sync bug、沒有漏欄、沒有型別降級、沒有漏列。**所有問題都在 cache 的上游(爬取與解析),cache 只是把上游的錯照抄一份。

## 不能信的地方(照嚴重度排)

### 1. BUG — 3 筆「轉讓股數」被黏成天文數字(會復發)

當**同一張申報同時用兩種轉讓方式**時(例如「一般交易 + 鉅額逐筆交易」),原始網頁把兩個股數放在同一格、中間用換行分隔。解析時 `cleanCell` 會把逗號**和空白一起**拿掉,於是兩個數字被黏成一個:

| 股票 | 日期 | 網頁原文(第 7 格) | 存進 DB 的值 |
|---|---|---|---|
| 2856 元富證券 | 2007-01-05 | `5,700,000 1,000,000` | **57,000,001,000,000**(57 兆) |
| 6994 富威電力 | 2026-06-29 | `656,000 6,360,000` | **6,560,006,360,000**(6.56 兆) |
| 8284 三竹 | 2026-05-13 | `580 253,000` | **580,253,000**(5.8 億) |

逐位重建驗證:`int('656000'+'6360000') == 6560006360000` 為 True(另兩筆同理)。

- **根因**:`reader/TradingReader.scala:1119` 用 `parseLong(cols(7))`,而 `cleanCell`(`TradingReader.scala:40-42`)裡的 `.replace(" ", "")` 把兩數之間的空白刪掉黏成一個。
- **這是一「類」不是孤例**:只要未來再有雙方式申報就會再中;而且 `cleanCell` 是所有數值欄共用的,任何「兩個數字擠同一格」的來源都會踩到。
- **影響**:目前僅 3 筆,但只要有人拿 `transfer_shares` 加總、排序或設門檻,這 3 筆會嚴重汙染結果。
- **修法**:雙方式申報應拆成兩筆(各自方式+各自股數)或加總,不能字元黏接;先用 6994 那列寫紅測試再驗綠;既有 3 筆用 `transfer_method LIKE '% %'` 鎖定重解析(PG 和 cache 兩邊都要改)。

### 2. REAL — 歷史 99% 是空的(已知待補,非新發現)

- 2026 年:03-31 ~ 07-17,70 個交易日(704 列)= 近期上線每天爬

### 3. SUSPECT — 2026 窗內 3 個「市場×日」漏爬 + 資料只到 07-17

在 2026 上線窗內,有 3 個(市場,日)連原始檔都沒有(疑似漏爬):

- twse **2026-04-09**、tpex **2026-04-29**、tpex **2026-05-28**

影響很小(內部人申報是事件型,多數日本來就零申報,這幾日 PG 對應 0~1 列),但是可補的真缺口。另外資料最新只到 `report_date` 2026-07-17,今天 07-23,爬蟲已數天沒跑(07-20/21/22 尚未抓)。

其中 2026-04-29、05-28 剛好也是 `C-foreign_holding_ratio` / `C-daily_quote` 記到的「假休市 sentinel」受害日——交易日曆讀 `daily_quote`,那幾天被假 sentinel 汙染,於是內部人連請求都沒送出。若確認同源,要先照那兩單的修法清假 sentinel、把交易日曆改成多來源共識,否則同一批日期會在所有日頻/事件表一起缺。

## 看起來怪、其實是真的(已排除,免得下一個人重查)

- **`transfer_shares` 有 570 筆是 0**:全部是**無每日轉讓限制**的方式——贈與 286、信託 234、洽特定人 29、轉讓私募股票 21(合計正好 570)。這些方式原始網頁那格本來就空白。非零的 201 筆正好是一般交易/鉅額逐筆/盤後定價。**真的,不是漏抓。**
- **`current_shares_trust` / `planned_shares_trust` 兩欄全表都是 0**:掃過全部 156 個原始檔,來源那兩格本來就都是 0(這段期間申報人持股都在「自有」、沒有「信託」部位),**不是欄位錯位**。但下游別把這兩欄當訊號(恆 0、零資訊)。
- **`declare_date` 每一列都等於 `report_date`**:這個端點是「當日新增事前申報」日報,它的「申報日期」欄本來就等於發布日(掃原始檔,不符的 0 筆)。真正的 lead-time(變更申報的原申報日)藏在被 reader 丟掉的註記格(例:`本單為變更115/06/08 之申報`),但那不是 `declare_date` 欄。Slick 註解說它「might lead by several days」對這個端點略微誤導,但**資料本身沒錯**。
- 其他基本掃描全清:負股數 0、未來日期 0、重複主鍵 0、market 只有 twse/tpex、股號格式全合法、轉讓方式都是合法列舉值。
