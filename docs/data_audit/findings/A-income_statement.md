# A-income_statement 稽核報告：綜合損益表原始檔 → DB 解析正確性

**結論：資料可以信任（OK）。** 綜合損益表原始檔裡的每一個數字欄位，都正確、完整、
單位無誤、正負號無誤地落到資料庫。唯一要放在心上的是一個「還沒發生但有前例」的看門
風險（Big5 編碼寫死），以及一個要用資料前先知道的邊界（cache 只放合併報表、沒有
2004 年以前）。

---

## 一句白話

我自己另外寫了一支程式，不碰系統原本的解析器，直接把原始 CSV 讀一遍，再拿去跟資料庫
一格一格對。橫跨 2006 到 2025、上市＋上櫃、個別＋合併、銀行／一般業／證券／保險／金控
各種不同表格式，總共 12 個季度、上百萬個數字——**沒有一格對不上**。台積電 2024 全年
營收在庫裡是 2,894,307,699（仟元，就是 2.894 兆），跟真實年報一模一樣；每股盈餘存的是
45.25（元），也對；2024 真的虧錢的公司（力積電、南亞科、彩晶…）淨利都正確存成負數。

## 為什麼這份資料的解析特別穩

這支解析器不是「照欄位位置硬讀第幾欄」，而是**照中文欄名對映**（讀成「欄名→值」的
字典再攤平）。所以台股原始檔最惡名昭彰的毛病——「MOPS 偷偷加一欄、後面全部錯位」——
在這裡結構上就不可能發生：新科目只會用它的新欄名自己長出來，不會把舊科目擠歪。實測
欄數從 20 欄長到 33 欄橫跨各年代，資料照樣一格不差。

## 兩件要記住的事（不是 bug，是邊界）

**1.（看門）Big5 編碼寫死——哪天 MOPS 改 UTF-8 會整檔炸掉。**
損益表解析器固定用 Big5 讀檔，沒有 UTF-8 的分支。隔壁的「月營收」解析器早就因為 MOPS
在 2012 年把檔案改成 UTF-8 而加了切換——**證明 MOPS 真的會這樣搞**。萬一損益表哪天也
改 UTF-8，中文欄名會變亂碼、排除清單失效，程式會在「拿公司代號」那行直接拋例外、那一
整檔匯不進去。**目前 2026 年以前的檔案全部還是 Big5，所以現在資料沒有錯**——這是提醒
下一個人盯著，不是現行故障。修法：比照月營收加編碼偵測，或匯入後檢查「公司代號欄在不
在」，不在就大聲報錯而不是裸拋。

**2.（用資料前必知）研究用的 cache 只有合併報表、而且沒有 2004 年以前。**
`is_progressive_raw`（研究層那份快取）只放合併報表，起點 2004 年（合併報表本來就 2004
第四季才開始）。資料庫本體 `concise_income_statement_progressive` 其實還有個別報表
（上市 1989 起、上櫃 1993 起）——但那些**不在** cache 裡。所以任何吃 cache 的回測，
天生看不到 2004 年以前的損益表。這是 cache 的設計取捨，不是解析錯。要更早的歷史就直接
查資料庫本體。

## 順帶釐清（都不是問題）

- **同一個 value 欄混單位**：貨幣科目是仟元、每股盈餘是元。下游程式必須看科目名才知道
  單位、不能一律相加——現行 margin 計算用「仟元÷仟元」比值自動抵消、EPS 另外取，處理
  正確。
- **只有存託憑證（DR/KY）外幣小表**的三個文字欄（換算匯率、換算匯率參考依據…）被丟掉：
  它們是文字不是數字，本來就塞不進「科目→數值」的表，全表才 115 列、不含任何損益科目，
  有界資訊損失。
- **19 個「檔案不存在」HTML 檔**：那是某市場某季某產業沒有公司（例如上櫃沒有銀行）時
  MOPS 回的空頁，被存成 .csv。解析器讀到它會乖乖產出 0 列、不會炸、不會汙染，該有的
  公司一家不少（上櫃 2004 個別 327 家，原始檔、資料庫完全對得上）。
