# C-market_index — cache 表 `market_index` 的一致性與缺漏

稽核日:2026-07-22 · 對象:`var/cache/cache.duckdb` 的 `market_index` vs PostgreSQL `public."index"`
可重跑腳本:`docs/data_audit/scripts/C-market_index/01~10_*.py`

---

## 一句話結論

**搬運沒問題,貨本身有問題。** cache 跟 PostgreSQL 一模一樣,769,751 列、6,837 個市場×日,
一位都不差,所以「cache 有沒有走樣」可以放心。真正不能信的是:**大盤指數(加權指數)這條線
有 8 天是別天的資料**——其中 3 天是週六、交易所根本沒開市,1 天存的是**十一天後**的收盤價
(前視);拿這條線算出來的「大盤當天漲跌」最誇張的一組是 2018-09-15 的 **-8.83%** 接 09-17 的
**+9.28%**,而交易所自己的紀錄是那兩天之間只跌了 0.36%。這些髒值還會在每次重建 cache 時被
二次加工:期貨基差有 3 天正負號被算反,2016-05-26 甚至算出「台指期對現貨溢價 7.3%」這種
不可能的數字。此外櫃買(上櫃)指數整整少了 **2024-06-27 到 08-12 這 31 個交易日**,連原始檔
都沒下載過。

**好消息**:現役實盤策略 Serenity 完全不碰這張表(全文搜尋 `src/quantlib/serenity/` 零命中),
所以這些髒污**不影響現在的下單決策**;受害的是研究線——Evergreen 的大盤 regime 判斷、
apex 的下跌市過濾、期貨基差因子。

## 1. 搬運層:cache 與 PostgreSQL 完全一致(OK)

| 檢查 | 結果 |
|---|---|
| 總列數 | pg 769,751 / cache 769,751,差 0 |
| 逐年 × 市場列數 | 29 格全等(twse 2009-2026、tpex 2016-2026) |
| 全史 (market,date) 三重指紋 | 6,837 天,mismatch 0、only_pg 0、only_cache 0 |
| NULL 分佈 | 兩邊 close_null 都是 1,578,change/pct 都是 0 |
| schema | PG 7 欄(含 `id`)→ cache 6 欄;`varchar→VARCHAR`、`date→DATE`、`double precision→DOUBLE`,零型別降級 |
| 投影 SQL | `research/cache_tables.py:46` 與 `src/quantlib/db.py:131-133` 逐字元相同(程式化比對回 True) |

指紋 = 每個 (market,date) 上算 `count(*)` + `sum(hash(6 欄)::HUGEINT)` + `bit_xor(hash(6 欄))`,
三個都相等才算過。抽樣逐欄比對(3 日期 × 5 指數,`pandas.DataFrame.equals`)也全 True。

被丟掉的唯一 PG 欄位是 `id`(Slick 的流水號代理鍵),無資訊損失。

## 2. 大盤指數這條線有 8 天是假的(BUG,947 列)

`market_index` 在 repo 裡幾乎只被拿來取一條線:`發行量加權股價指數`(加權指數)。
這條線上有 8 天存的是**別的日期的資料**——整天每一檔指數的收盤與漲跌都與來源日一字不差。

用**交易所自己的資料**(TWSE 每日市場成交資訊 FMTQIK 月報,2026-07-22 實抓)當真值對照:

| 受汙染日 | cache 存的收盤 | 交易所真值 | 差 | 備註 |
|---|---|---|---|---|
| 2015-08-29(六) | 8,257.32 | **當天沒開市** | — | 整天 93 列是 2015-12-18 的複本 |
| 2016-05-26 | 7,811.18 | 8,394.12 | **-582.94** | 是 2016-01-18 的複本 |
| 2017-08-02 | 10,506.52 | 10,519.27 | -12.75 | 是 2017-12-18 的複本 |
| 2018-08-04(六) | 10,995.39 | **當天沒開市** | — | 是 2018-07-24 的複本 |
| 2018-09-15(六) | 9,908.69 | **當天沒開市** | — | 是 2017-03-17 的複本 |
| 2018-10-03 | 11,087.47 | 10,863.94 | **+223.53** | 是 2018-06-15 的複本 |
| 2019-07-05 | 10,886.05 | 10,785.73 | +100.32 | 是 **2019-07-16** 的複本 → **前視** |
| 2019-09-25 | 10,634.85 | 10,873.69 | **-238.84** | 是 2019-09-02 的複本 |

### 拿這條線算「大盤今天漲跌」會錯成什麼樣

| 前一日 → 受汙染日 → 次一日 | cache 算出來 | 真實 |
|---|---|---|
| 2018-09-14 → 09-15 → 09-17 | **-8.83%** 然後 **+9.28%** | 那兩天之間只有 **-0.36%**(09-15 是週六,根本不存在) |
| 2016-05-25 → 05-26 → 05-27 | -6.97% 然後 +8.35% | -0.02% 然後 +0.83% |
| 2019-09-24 → 09-25 → 09-26 | -2.59% 然後 +2.23% | -0.41% 然後 -0.02% |
| 2018-10-02 → 10-03 → 10-04 | +1.54% 然後 -3.32% | -0.51% 然後 -1.33% |
| 2015-08-28 → 08-29 → 08-31 | 憑空多一根 +2.97%,再 -1.00% | 真實只有一根 +1.94% |

全表 4,303 天裡,最多 16 個「大盤日報酬」是假的(每個幽靈日汙染當天與隔天各一根)。
**這正好是 regime / 下跌市過濾器最敏感的那種輸入**——一根假的 -8.8% 足以把整段時間
標成空頭。

### 為什麼判得出來(不是猜的)

三種互相獨立的方法收斂到同一組日期:

1. **整日內容指紋撞號**(`04_sample_and_anomaly.py` 第 ③ 段):同市場兩個不同日期,
   整天 (name, close, change) 完全相同 → 抓到 6 對。
2. **交集名稱全等**(`05_phantom_probe.py` 第 A 段):放寬成「共同指數名 ≥50 且值全同」
   → 補齊到 **8 對**(2016-05-26 與 2019-07-05 是部分複製,嚴格指紋抓不到)。
3. **語意一致性**(`04` 第 ④ 段):`|今日收盤 − 昨日收盤 − 今日漲跌點| > 0.02` 的比例,
   767,732 個可檢查列裡 2,931 列不一致,集中在這 8 天與其次一日。

再加上**交易所第一手覆核**:FMTQIK 月報顯示 104/08 沒有 08/29、107/08 沒有 08/04、
107/09 沒有 09/15(三個週六確實沒開市),而 105/05/26 = 8,394.12、106/08/02 = 10,519.27、
107/10/03 = 10,863.94、108/07/05 = 10,785.73、108/09/25 = 10,873.69(五個平日的真值)。

原始檔佐證前視:`data/index/twse/2019/2019_7_5.csv` 檔頭寫「108年07月05日」,
內容與 `2019_7_16.csv` 一字不差。

## 3. 髒 TAIEX 會被 cache 二次加工,汙染期貨基差(BUG)

`src/quantlib/futures/taifex.py:216-217` 在**建 cache 時現算**基差:
`tx_spot_basis = 期貨結算價 − taiex_close`。5 個平日幽靈日都落在期貨因子表裡:

| 日期 | 基差(存) | 基差%(存) | 基差(真) | 基差%(真) |
|---|---|---|---|---|
| 2016-05-26 | +571.82 | **+7.32%** | -11.12 | -0.13% |
| 2017-08-02 | -53.52 | -0.51% | -66.27 | -0.63% |
| 2018-10-03 | -233.47 | **-2.11%** | -9.94 | -0.09% |
| 2019-07-05 | -168.05 | -1.54% | -67.73 | -0.63% |
| 2019-09-25 | +206.15 | **+1.94%** | -32.69 | -0.30% |

近月台指期對現貨的基差正常在 ±1% 內。**3 天正負號被算反**,2016-05-26 更被算成
7.3% 溢價——物理上不可能。重點是:`taifex_futures_daily_factors` 不是 PG 的複本,
是 cache 自己生的衍生表,所以**每重建一次 cache,髒污就被重新加工擴散一次**。

## 5. 加權指數在 2019-04-29 有一天洞——因為那天它叫別的名字(BUG)

2019-04-29 是 TWSE 換指數名冊的那一天,當天的檔案把大盤指數印成 **`加權股價指數`**
(少了「發行量」三個字),隔天起才固定成 `發行量加權股價指數`。

值本身是對的:`加權股價指數` close=10,939.06、change=-13.41,而 2019-04-26 收 10,952.47,
10,952.47 − 13.41 = 10,939.06 ✓。

但 repo 裡所有消費者都寫 `WHERE name = '發行量加權股價指數'`
(`src/quantlib/evergreen/{ev26_engine.py:60,ev46_downmkt.py:29,ev51_regime_defs.py:27,ev49b_regime_real.py:49}`、

**這天會被靜靜跳過**,於是 2019-04-30 的「日報酬」實際上橫跨兩天。

同一天還有 4 個只出現過一次的名字:`加權股價指數`、`加權股價報酬指數`、`寶島指數`、`寶島報酬指數`。

## 7. 從 A 維原封傳進來的兩件事(cache 忠實,但值本身有問題)

- **1,578 列「當天未公布」被寫成漲跌 0**:`close` 是 NULL(正確)但 `change=0`、

這兩條的根因與修法都在 A 維;cache 側只能等 PG 修好後重跑 `research/cache_tables.py`。

## 8. cache 落後齊備日兩個交易日,而且正在表間錯位(SUSPECT)

```
latest_complete_trading_day() = 2026-07-21
market_index      max(date) = 2026-07-17   ← 落後
daily_quote       max(date) = 2026-07-20
daily_trading_details       = 2026-07-20
stock_per_pbr               = 2026-07-20
```

原因:`index` 不在 Python 直寫路徑的日頻源清單裡(`src/quantlib/crawl/update.py:27` 的
`DAILY_SOURCES` 只有 daily_quote / daily_trading_details / stock_per_pbr),只能靠 Scala
`Main update` 抓進 PG 再全砍重建 cache;最後一次 Scala 爬蟲是 2026-07-19 21:55
(`data/index/twse/2026/2026_7_17.csv` 的 mtime)。

這正是 CLAUDE.md 記載 2026-07-15 事故的形態(表間日期錯位 → 閘門 fail-closed 靜靜砍光候選)。
與 `C-daily_quote` / `C-margin_transactions` 是同一條根因。

## 10. 建議修法(不在本單位執行)

依序做,順序不能反:

1. **先刪汙染列與哨兵檔,再補抓**(reader 以「(market, 檔名) 已在 DB 就跳過」去重,不刪就補不進去):
   ```sql
   DELETE FROM index WHERE market='twse' AND date IN
     ('2015-08-29','2016-05-26','2017-08-02','2018-08-04',
      '2018-09-15','2018-10-03','2019-07-05','2019-09-25');
   ```
   ```bash
   # 3 個週六改回 0-byte sentinel,其餘刪檔重抓
   rm data/index/twse/{2016/2016_5_26,2017/2017_8_2,2018/2018_10_3,2019/2019_7_5,2019/2019_9_25}.csv
   rm data/index/twse/{2009/2009_12_12,2026/2026_3_12}.csv        # 0-byte 哨兵擋住重抓
   ```
2. **補抓缺日**
   - TWSE:`https://www.twse.com.tw/rwd/zh/afterTrading/MI_INDEX?response=csv&type=IND&date=YYYYMMDD`
     → 2009-12-12、2026-03-12、2026-05-28
   - TPEx:`https://www.tpex.org.tw/www/zh-tw/afterTrading/indexSummary?response=csv&date=YYYYMMDD`
     → 2024-06-27 ~ 2024-08-12(31 天)、2026-05-28
   - 2016-09-10 tpex 來源端當天就沒公布(122 bytes),補不回來,只能記錄。
3. **重跑** `sbt "runMain Main read index"` + `uv run python research/cache_tables.py`。
4. **根因守護**(防復發,先紅後綠):
   - `IndexSetting.validate` 除了比對檔頭日期,再驗「本檔加權指數的 `close − change` 是否
     等於 DB 中前一交易日的 close(容差 0.02)」,對不上即 `[deferred]` 刪檔重抓。
     只驗檔頭日期擋不住 TWSE 的靜默 fallback(同一失效模式已在 A-daily_quote / A-index 出現)。
   - `getDatesOfExistFiles` 把 0-byte 以外的「≤8 bytes」與「已知空回應以外的異常小檔」視為未下載。
   - 加測試鎖死:`market_index` 的 (market,date) 集合 ⊇ 同市場同區間 `daily_quote` 的 (market,date) 集合。
   - 加測試鎖死:每個 (market,date) 的指數名集合與前一交易日的 Jaccard 相似度 ≥ 0.9,
     低於即判「舊名冊/半殘檔」→ `[deferred]` 重抓(這條會直接抓到 2026-02-26 / 2026-03-11)。
5. **消費端**:所有取加權指數的地方改成 `name IN ('發行量加權股價指數','加權股價指數')`
   或先做名稱正規化表,才不會在 2019-04-29 掉一天。
6. **新鮮度**:把 index / margin / sbl / foreign 納入 `src/quantlib/crawl/update.py` 的 `DAILY_SOURCES`,
   或在每日 loop 明確要求先跑 Scala `Main update`;在此之前 `ensure_fresh_cache` 的擋門不可放寬。
7. **cache 唯一性**:建表後補 `CREATE UNIQUE INDEX ON market_index(market, date, name)`
   (PG 有 `idx_Index_market_date_name` UNIQUE,cache 目前只有非唯一的 `idx_market_index_date`;
   實測重複 0 筆,但唯一性目前只靠「每次砍掉重建」的自律維持)。
